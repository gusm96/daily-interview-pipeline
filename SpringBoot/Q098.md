## Spring Bean 스코프

면접관님, Spring Bean 스코프는 Spring 컨테이너가 Bean의 인스턴스를 생성하고 관리하는 방식을 정의하는 중요한 개념입니다. 각 스코프는 특정 사용 시나리오에 적합한 특징을 가지며, 이를 이해하고 적절히 활용하는 것이 애플리케이션의 성능과 안정성에 큰 영향을 미칩니다.

### 1. Singleton 스코프

*   **핵심 정의:** Spring 컨테이너당 단 하나의 Bean 인스턴스만 생성하고 관리합니다. 모든 요청에서 동일한 인스턴스를 공유합니다.
*   **동작 원리:** 컨테이너가 초기화될 때 또는 해당 Bean이 처음 요청될 때 인스턴스를 생성하고, 이후 모든 요청에 대해 캐시된 동일한 인스턴스를 반환합니다.
*   **실무에서 주의할 점:**
    *   **상태 없음(Stateless) 지향:** 여러 스레드에서 동시에 접근하므로, Bean 내부에 변경 가능한 상태(Mutable State)를 가지지 않도록 설계해야 합니다. 상태를 가질 경우 동시성 문제가 발생할 수 있습니다.
    *   **공유 자원:** 데이터베이스 연결 풀, 서비스 계층의 비즈니스 로직 객체 등 애플리케이션 전반에 걸쳐 공유되어야 하는 무상태 객체에 적합합니다.
    *   **메모리 효율성:** 하나의 인스턴스만 존재하므로 메모리 사용량이 적습니다.

### 2. Prototype 스코프

*   **핵심 정의:** Bean을 요청할 때마다 Spring 컨테이너가 새로운 Bean 인스턴스를 생성합니다.
*   **동작 원리:** `getBean()` 메서드가 호출될 때마다 새로운 인스턴스를 생성하여 반환합니다. 컨테이너는 생성까지만 관여하고, 이후 Bean의 생명주기 관리는 클라이언트에게 위임됩니다.
*   **실무에서 주의할 점:**
    *   **상태 유지(Stateful) 가능:** 각 요청마다 새로운 인스턴스가 생성되므로, Bean 내부에 변경 가능한 상태를 가질 수 있습니다.
    *   **자원 관리:** 컨테이너가 Bean의 소멸을 관리하지 않으므로, 개발자가 직접 자원을 해제하거나 GC에 맡겨야 합니다. 특히 자원을 많이 사용하는 객체라면 주의해야 합니다.
    *   **활용 사례:** 매번 다른 데이터를 담아야 하는 임시 객체, 특정 작업에만 사용되고 버려지는 객체 등에 적합합니다.

### 3. Request 스코프 (웹 환경 전용)

*   **핵심 정의:** 하나의 HTTP 요청이 처리되는 동안만 Bean 인스턴스가 유지됩니다. 요청이 끝나면 소멸됩니다.
*   **동작 원리:** HTTP 요청이 들어올 때마다 새로운 인스턴스를 생성하고, 해당 요청이 완료되면 인스턴스를 소멸시킵니다.
*   **실무에서 주의할 점:**
    *   **요청별 데이터 관리:** 특정 HTTP 요청과 관련된 데이터를 저장하고 공유하는 데 유용합니다. 예를 들어, 사용자 로그인 정보, 요청 추적 ID 등을 저장할 수 있습니다.
    *   **프록시 모드:** Singleton Bean이 Request 스코프 Bean을 주입받을 경우, 직접 주입받는 것이 아니라 프록시 객체를 주입받아 실제 Request 스코프 Bean에 접근하도록 설정해야 합니다. (`@Scope(value = "request", proxyMode = ScopedProxyMode.TARGET_CLASS)`)
    *   **활용 사례:** 웹 애플리케이션에서 각 요청마다 고유하게 유지되어야 하는 사용자 정보, 트랜잭션 컨텍스트 등에 활용됩니다.

### 4. Session 스코프 (웹 환경 전용)

*   **핵심 정의:** 하나의 HTTP 세션이 유지되는 동안 Bean 인스턴스가 유지됩니다. 세션이 종료되면 소멸됩니다.
*   **동작 원리:** HTTP 세션이 생성될 때 새로운 인스턴스를 생성하고, 해당 세션이 만료되거나 무효화되면 인스턴스를 소멸시킵니다.
*   **실무에서 주의할 점:**
    *   **사용자별 데이터 관리:** 특정 사용자 세션과 관련된 데이터를 저장하고 공유하는 데 유용합니다. 예를 들어, 장바구니 정보, 사용자 설정 등을 저장할 수 있습니다.
    *   **프록시 모드:** Request 스코프와 마찬가지로 Singleton Bean이 Session 스코프 Bean을 주입받을 경우 프록시 모드를 사용해야 합니다. (`@Scope(value = "session", proxyMode = ScopedProxyMode.TARGET_CLASS)`)
    *   **세션 관리:** 세션의 과도한 사용은 서버 메모리 부하를 증가시킬 수 있으므로, 꼭 필요한 데이터만 저장하고 세션 타임아웃을 적절히 설정해야 합니다.
    *   **활용 사례:** 웹 애플리케이션에서 사용자 로그인 상태, 장바구니, 개인화된 설정 등 세션 동안 유지되어야 하는 정보에 활용됩니다.

### 특정 스코프 선택 시 고려 사항 및 활용 전략

*   **상태 유무:** Bean이 변경 가능한 상태를 가지는지 여부가 가장 중요한 판단 기준입니다.
    *   **무상태(Stateless):** Singleton 스코프를 기본으로 고려합니다.
    *   **상태 유지(Stateful):** Prototype, Request, Session 스코프를 고려하며, 상태의 범위(요청별, 세션별)에 따라 선택합니다.
*   **공유 범위:** Bean 인스턴스가 얼마나 넓은 범위에서 공유되어야 하는지를 고려합니다.
    *   **애플리케이션 전역:** Singleton
    *   **요청별:** Request
    *   **세션별:** Session
    *   **매번 새로운 인스턴스:** Prototype
*   **성능 및 메모리:**
    *   Singleton은 메모리 효율적이지만, 동시성 문제를 주의해야 합니다.
    *   Prototype, Request, Session은 새로운 인스턴스 생성 비용과 메모리 사용량이 증가할 수 있으므로, 꼭 필요한 경우에만 사용합니다.
*   **생명주기 관리:**
    *   Singleton은 Spring 컨테이너가 완벽하게 관리합니다.
    *   Prototype은 개발자가 소멸을 직접 관리해야 할 수 있습니다.
    *   Request, Session은 웹 컨테이너의 생명주기에 따라 관리됩니다.

실무에서는 대부분의 Bean을 **Singleton 스코프**로 사용하며, 무상태로 설계하여 동시성 문제를 방지하는 것이 일반적인 전략입니다. 특정 요청이나 사용자 세션에 종속적인 상태를 관리해야 할 때만 Request나 Session 스코프를 사용하고, 매번 독립적인 객체가 필요한 경우에 Prototype 스코프를 고려합니다. 각 스코프의 특징을 명확히 이해하고 애플리케이션의 요구사항에 맞춰 신중하게 선택하는 것이 중요합니다.
