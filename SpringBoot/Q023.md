## Spring `@Transactional` 모범 답안

Spring의 `@Transactional` 어노테이션은 선언적 트랜잭션 관리를 위한 핵심 기능입니다. 내부 동작 방식, 전파 및 격리 옵션, 그리고 주의사항에 대해 설명드리겠습니다.

### 1. `@Transactional` 내부 동작 방식

`@Transactional`은 Spring AOP(Aspect-Oriented Programming)를 기반으로 동작합니다.

1.  **AOP 프록시 생성:** Spring 컨테이너는 `@Transactional`이 붙은 빈을 발견하면, 해당 빈의 원본 객체 대신 **프록시 객체**를 생성합니다. 이 프록시는 원본 객체를 감싸는 형태로 존재합니다.
2.  **트랜잭션 인터셉터:** 이 프록시 내부에는 `TransactionInterceptor`라는 AOP 어드바이스가 연결되어 있습니다.
3.  **메서드 호출 가로채기:** 클라이언트가 `@Transactional` 메서드를 호출하면, 실제로는 원본 객체의 메서드가 아닌 **프록시 객체의 메서드**가 호출됩니다. 프록시는 이 호출을 가로채 `TransactionInterceptor`로 전달합니다.
4.  **트랜잭션 매니저 위임:** `TransactionInterceptor`는 `PlatformTransactionManager`에게 트랜잭션 관련 작업을 위임합니다.
    *   **트랜잭션 시작:** 메서드 실행 전, `PlatformTransactionManager`는 새로운 트랜잭션을 시작하거나 기존 트랜잭션에 참여합니다. 이 과정에서 데이터베이스 커넥션을 얻고, `setAutoCommit(false)`를 설정하는 등의 작업을 수행합니다.
    *   **메서드 실행:** 실제 비즈니스 로직이 담긴 원본 객체의 메서드가 실행됩니다.
    *   **트랜잭션 커밋/롤백:** 메서드 실행이 성공적으로 완료되면 `PlatformTransactionManager`는 트랜잭션을 **커밋**합니다. 만약 예외가 발생하면 트랜잭션을 **롤백**합니다. (기본적으로 RuntimeException과 Error는 롤백, Checked Exception은 커밋)
5.  **리소스 정리:** 트랜잭션 종료 후, `PlatformTransactionManager`는 데이터베이스 커넥션 반환 등 리소스를 정리합니다.

`PlatformTransactionManager`는 JDBC, JPA, JMS 등 다양한 기술 스택에 대한 추상화 계층을 제공하여, 개발자가 특정 트랜잭션 기술에 종속되지 않고 일관된 방식으로 트랜잭션을 관리할 수 있도록 돕습니다.

### 2. 전파(Propagation) 옵션

전파 옵션은 이미 진행 중인 트랜잭션이 있을 때, 새로운 트랜잭션이 어떻게 동작할지 정의합니다.

*   **`REQUIRED` (기본값):**
    *   **동작:** 이미 진행 중인 트랜잭션이 있으면 해당 트랜잭션에 참여하고, 없으면 새로운 트랜잭션을 시작합니다.
    *   **예시:**
        ```java
        @Service
        public class UserService {
            @Transactional(propagation = Propagation.REQUIRED) // 트랜잭션 A
            public void createUserAndLog(User user) {
                userRepository.save(user); // 트랜잭션 A에 참여
                logService.logAction("User created"); // 트랜잭션 A에 참여
            }
        }

        @Service
        public class LogService {
            @Transactional(propagation = Propagation.REQUIRED) // 트랜잭션 A에 참여
            public void logAction(String action) {
                logRepository.save(new Log(action));
            }
        }
        ```
        `createUserAndLog`가 호출될 때 트랜잭션 A가 시작되고, `logAction`은 트랜잭션 A에 참여합니다. `createUserAndLog`에서 예외 발생 시, `userRepository.save`와 `logRepository.save` 모두 롤백됩니다.

*   **`REQUIRES_NEW`:**
    *   **동작:** 항상 새로운 트랜잭션을 시작합니다. 이미 진행 중인 트랜잭션이 있으면 해당 트랜잭션을 잠시 중단시키고 새로운 트랜잭션을 시작합니다. 새로운 트랜잭션이 완료되면 중단된 트랜잭션을 재개합니다.
    *   **예시:**
        ```java
        @Service
        public class UserService {
            @Transactional(propagation = Propagation.REQUIRED) // 트랜잭션 A
            public void createUserAndLog(User user) {
                userRepository.save(user); // 트랜잭션 A에 참여
                try {
                    logService.logActionSeparately("User created"); // 트랜잭션 B (새로운 트랜잭션)
                } catch (Exception e) {
                    // 로그 저장 실패는 사용자 생성에 영향을 주지 않음
                }
            }
        }

        @Service
        public class LogService {
            @Transactional(propagation = Propagation.REQUIRES_NEW) // 항상 새로운 트랜잭션 B 시작
            public void logActionSeparately(String action) {
                logRepository.save(new Log(action));
            }
        }
        ```
        `createUserAndLog`가 호출될 때 트랜잭션 A가 시작됩니다. `logActionSeparately`가 호출되면 트랜잭션 A는 잠시 중단되고 트랜잭션 B가 시작됩니다. `logActionSeparately`에서 예외가 발생하여 트랜잭션 B가 롤백되더라도, 트랜잭션 A는 영향을 받지 않고 계속 진행될 수 있습니다.

### 3. 격리(Isolation) 옵션

격리 옵션은 여러 트랜잭션이 동시에 실행될 때, 한 트랜잭션이 다른 트랜잭션의 변경 내용을 얼마나 볼 수 있는지 정의합니다. 데이터베이스의 격리 수준과 동일하며, 일반적으로 데이터베이스 기본 설정을 따릅니다.

*   **`DEFAULT` (기본값):**
    *   **동작:** 데이터베이스의 기본 격리 수준을 사용합니다. 대부분의 데이터베이스는 `READ_COMMITTED` 또는 `REPEATABLE_READ`를 기본값으로 사용합니다.
*   **`READ_UNCOMMITTED`:**
    *   **동작:** 다른 트랜잭션이 커밋하지 않은 변경 내용도 읽을 수 있습니다. **Dirty Read** 문제가 발생할 수 있어 거의 사용되지 않습니다.
*   **`READ_COMMITTED`:**
    *   **동작:** 다른 트랜잭션이 커밋한 데이터만 읽을 수 있습니다. **Dirty Read**는 방지하지만, 한 트랜잭션 내에서 같은 쿼리를 두 번 실행했을 때 다른 결과가 나올 수 있는 **Non-Repeatable Read** 문제가 발생할 수 있습니다.
    *   **예시:**
        ```java
        // 트랜잭션 A (격리 수준: READ_COMMITTED)
        @Transactional(isolation = Isolation.READ_COMMITTED)
        public void processOrder(Long orderId) {
            Order order1 = orderRepository.findById(orderId).orElseThrow(); // 1. 주문 상태: PENDING

            // (다른 트랜잭션 B가 이 시점에 orderId의 상태를 PENDING -> SHIPPED로 변경하고 커밋)

            Order order2 = orderRepository.findById(orderId).orElseThrow(); // 2. 주문 상태: SHIPPED
            // order1과 order2의 상태가 다를 수 있음 (Non-Repeatable Read)
        }
        ```
*   **`REPEATABLE_READ`:**
    *   **동작:** 한 트랜잭션 내에서 같은 쿼리를 여러 번 실행해도 항상 동일한 결과를 보장합니다. **Non-Repeatable Read**를 방지하지만, 새로운 레코드가 추가되는 **Phantom Read** 문제가 발생할 수 있습니다. MySQL의 기본 격리 수준입니다.
    *   **예시:**
        ```java
        // 트랜잭션 A (격리 수준: REPEATABLE_READ)
        @Transactional(isolation = Isolation.ISOLATION_REPEATABLE_READ)
        public void countUsersByAge(int age) {
            long count1 = userRepository.countByAge(age); // 1. 20세 사용자 5명

            // (다른 트랜잭션 B가 이 시점에 20세 사용자 1명을 추가하고 커밋)

            long count2 = userRepository.countByAge(age); // 2. 여전히 20세 사용자 5명 (Non-Repeatable Read 방지)

            // (하지만, 다른 쿼리(예: findAll)를 실행하면 새로 추가된 사용자가 보일 수 있음 - Phantom Read)
        }
        ```
*   **`SERIALIZABLE`:**
    *   **동작:** 가장 높은 격리 수준으로, 모든 동시성 문제를 방지합니다. 트랜잭션이 순차적으로 실행되는 것처럼 동작하여 **Dirty Read, Non-Repeatable Read, Phantom Read** 모두 방지합니다. 하지만 동시성이 크게 저하되어 성능 문제가 발생할 수 있으므로 신중하게 사용해야 합니다.

### 4. `@Transactional` 사용 시 주의할 점

1.  **Self-invocation (자가 호출) 문제:**
    *   **문제:** `@Transactional`이 적용된 클래스 내부에서 다른 `@Transactional` 메서드를 직접 호출할 경우, 트랜잭션 프록시가 개입하지 않아 트랜잭션이 적용되지 않습니다.
    *   **원인:** Spring AOP는 프록시를 통해 동작합니다. `this.someTransactionalMethod()`와 같이 내부에서 직접 호출하면 프록시를 거치지 않고 원본 객체의 메서드를 직접 호출하게 됩니다.
    *   **예시:**
        ```java
        @Service
        public class MyService {
            @Transactional // 트랜잭션 A
            public void outerMethod() {
                // ...
                this.innerMethod(); // 문제 발생! innerMethod의 @Transactional이 적용되지 않음
                // ...
            }

            @Transactional(propagation = Propagation.REQUIRES_NEW) // 트랜잭션 B (적용 안 됨)
            public void innerMethod() {
                // ...
            }
        }
        ```
    *   **해결책:**
        *   **별도의 서비스 클래스로 분리:** `@Transactional` 메서드를 다른 서비스 클래스로 분리하여 주입받아 사용합니다.
        *   **`AopContext.currentProxy()` 사용 (권장하지 않음):** 현재 프록시 객체를 직접 얻어 호출할 수 있지만, 코드 복잡성을 증가시키고 AOP 의존성을 높여 권장되지 않습니다.
        *   **`@Transactional`을 호출하는 메서드에만 적용:** 내부 호출되는 메서드에 `@Transactional`을 적용하지 않고, 외부에서 호출되는 메서드에만 적용하여 하나의 트랜잭션으로 묶는 것을 고려합니다.

2.  **Checked Exception 롤백 문제:**
    *   기본적으로 `@Transactional`은 `RuntimeException`과 `Error` 발생 시에만 롤백을 수행합니다. `IOException`과 같은 Checked Exception은 기본적으로 롤백되지 않고 커밋됩니다.
    *   **해결책:** `rollbackFor` 또는 `noRollbackFor` 속성을 사용하여 롤백 정책을 명시적으로 지정해야 합니다.
        ```java
        @Transactional(rollbackFor = IOException.class)
        public void processFile() throws IOException {
            // ...
        }
        ```

3.  **프록시 방식의 한계 (private, final 메서드):**
    *   Spring AOP는 기본적으로 CGLIB 프록시를 사용하여 클래스를 상속받아 프록시를 생성합니다. 따라서 `private` 메서드나 `final` 메서드에는 `@Transactional`을 적용할 수 없습니다. (JDK 동적 프록시는 인터페이스 기반)
    *   **해결책:** `private`이나 `final` 메서드에 트랜잭션이 필요하다면, 해당 메서드를 `public`으로 변경하거나 별도의 클래스로 분리해야 합니다.

4.  **트랜잭션 범위의 적절성:**
    *   트랜잭션의 범위는 너무 넓지도, 너무 좁지도 않게 적절히 설정해야 합니다. 너무 넓으면 불필요한 락으로 동시성 문제가 발생할 수 있고, 너무 좁으면 원자성이 깨질 수 있습니다. 일반적으로 비즈니스 로직의 한 단위(Service 계층)에 적용하는 것이 좋습니다.

`@Transactional`은 Spring 애플리케이션에서 트랜잭션 관리를 매우 편리하게 해주지만, 그 내부 동작 방식과 옵션, 그리고 주의사항을 정확히 이해하고 사용해야 예상치 못한 문제를 방지하고 안정적인 애플리케이션을 개발할 수 있습니다.
