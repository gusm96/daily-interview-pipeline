## Spring AOP(Aspect-Oriented Programming)

### 1. 핵심 개념

Spring AOP는 횡단 관심사(Cross-cutting Concerns)를 모듈화하여 애플리케이션의 핵심 비즈니스 로직과 분리하는 프로그래밍 패러다임입니다. 이를 통해 코드의 응집도를 높이고 유지보수성을 향상시킵니다.

*   **Aspect (애스펙트)**: 여러 클래스에 걸쳐 공통적으로 나타나는 횡단 관심사를 모듈화한 것입니다. Advice와 Pointcut의 조합으로 구성됩니다.
*   **Advice (어드바이스)**: 특정 Joinpoint에서 Aspect가 수행하는 실제 동작입니다. (예: 메서드 호출 전 로깅, 예외 발생 시 트랜잭션 롤백)
    *   **Before Advice**: Joinpoint 실행 전에 실행됩니다.
    *   **After Returning Advice**: Joinpoint가 성공적으로 실행된 후 실행됩니다.
    *   **After Throwing Advice**: Joinpoint에서 예외가 발생했을 때 실행됩니다.
    *   **After (Finally) Advice**: Joinpoint 실행 결과와 상관없이 실행됩니다.
    *   **Around Advice**: Joinpoint 실행 전후를 모두 제어하며, Joinpoint 실행 여부도 결정할 수 있습니다.
*   **Pointcut (포인트컷)**: Advice가 적용될 Joinpoint를 정의하는 표현식입니다. 정규 표현식이나 AspectJ 문법을 사용하여 특정 메서드나 클래스를 지정합니다.
*   **Joinpoint (조인포인트)**: Advice가 적용될 수 있는 애플리케이션 실행 지점입니다. Spring AOP에서는 메서드 실행 지점만 지원합니다.
*   **Weaving (위빙)**: Aspect가 핵심 코드에 적용되는 과정입니다. Spring AOP는 런타임에 프록시를 사용하여 위빙을 수행합니다.

### 2. Spring AOP 동작 원리 (프록시 기반)

Spring AOP는 기본적으로 **프록시 기반**으로 동작합니다.

*   Spring 컨테이너는 AOP 설정에 따라 Aspect가 적용될 대상 객체(Target Object)에 대한 프록시 객체를 생성합니다.
*   클라이언트가 대상 객체를 호출하면, 실제로는 프록시 객체가 호출됩니다.
*   프록시 객체는 호출된 메서드가 Pointcut에 해당하는지 확인하고, 해당한다면 정의된 Advice를 실행한 후 실제 대상 객체의 메서드를 호출합니다.
*   Spring AOP는 기본적으로 JDK 동적 프록시를 사용하며, 인터페이스가 없는 클래스에 AOP를 적용할 때는 CGLIB 프록시를 사용합니다.

### 3. 실무 활용 시나리오

*   **로깅**: 모든 서비스 계층 메서드 호출 전후에 로그를 남겨 실행 시간, 파라미터, 반환 값 등을 기록합니다.
*   **트랜잭션 관리**: 서비스 메서드에 `@Transactional` 어노테이션을 적용하여 선언적으로 트랜잭션을 시작하고 커밋/롤백을 처리합니다.
*   **보안**: 특정 메서드 실행 전에 사용자 인증 및 권한 검사를 수행하여 인가되지 않은 접근을 차단합니다.
*   **성능 모니터링**: 메서드 실행 시간을 측정하여 병목 지점을 파악하고 성능 최적화에 활용합니다.
*   **예외 처리**: 특정 예외 발생 시 공통적인 예외 처리 로직(예: 에러 로깅, 사용자 친화적인 메시지 반환)을 적용합니다.

### 4. AOP 사용 시 주의할 점

*   **과도한 사용 지양**: AOP는 강력하지만, 너무 많은 곳에 적용하면 코드의 흐름을 추적하기 어렵게 만들고 디버깅을 복잡하게 할 수 있습니다. 핵심 비즈니스 로직과 명확히 분리되는 횡단 관심사에만 적용하는 것이 좋습니다.
*   **프록시 기반의 한계**: Spring AOP는 프록시 기반이므로, 대상 객체 내부에서 자기 자신을 호출하는 메서드(self-invocation)에는 AOP가 적용되지 않습니다. 이 경우 프록시 객체를 직접 주입받아 사용하거나, AspectJ와 같은 컴파일 타임 위빙을 고려해야 합니다.
*   **성능 오버헤드**: 프록시 생성 및 Advice 실행으로 인해 미미한 성능 오버헤드가 발생할 수 있습니다. 일반적으로 큰 문제는 아니지만, 매우 성능에 민감한 시스템에서는 고려할 필요가 있습니다.
*   **테스트 용이성**: AOP가 적용된 코드를 테스트할 때, Advice의 동작을 고려해야 합니다. Mocking 등을 통해 Advice의 영향을 최소화하거나, 통합 테스트를 통해 실제 동작을 검증하는 것이 좋습니다.
*   **AspectJ와의 차이점 이해**: Spring AOP는 AspectJ의 부분 집합이며, AspectJ는 더 강력한 기능을 제공합니다 (예: 필드 접근, 생성자 호출 등 다양한 Joinpoint 지원, 컴파일 타임 위빙). Spring AOP는 Spring 컨테이너 내에서 AOP를 쉽게 적용하기 위한 경량 솔루션임을 이해해야 합니다.
