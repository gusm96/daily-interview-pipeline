## Spring Data JPA 커스텀 리포지토리 구현 및 활용

### 1. 핵심 정의
Spring Data JPA의 커스텀 리포지토리는 `JpaRepository`와 같은 기본 인터페이스만으로는 해결하기 어려운 복잡한 쿼리, 동적 쿼리, 또는 특정 비즈니스 로직을 포함하는 데이터 접근 로직을 직접 구현하기 위한 방법입니다. 이를 통해 Spring Data JPA의 추상화 계층을 유지하면서도 유연성을 확보할 수 있습니다.

### 2. 동작 원리
커스텀 리포지토리는 크게 세 단계로 구현됩니다.

*   **커스텀 인터페이스 정의:** `JpaRepository`를 상속받는 기존 리포지토리 인터페이스와 동일한 패키지에 커스텀 메서드를 정의하는 인터페이스를 생성합니다. 이 인터페이스의 이름은 일반적으로 `[엔티티명]RepositoryCustom`과 같이 명명합니다.
*   **커스텀 구현체 정의:** 위에서 정의한 커스텀 인터페이스를 구현하는 클래스를 생성합니다. 이 클래스의 이름은 일반적으로 `[엔티티명]RepositoryImpl`과 같이 명명하며, `@Repository` 어노테이션은 붙이지 않습니다. 이 구현체 내부에서 `EntityManager`나 `JdbcTemplate`, `Querydsl` 등을 활용하여 복잡한 쿼리 로직을 작성합니다.
*   **기존 리포지토리 인터페이스 확장:** 마지막으로, 기존의 `JpaRepository`를 상속받는 리포지토리 인터페이스가 앞서 정의한 커스텀 인터페이스를 상속받도록 확장합니다. Spring Data JPA는 명명 규칙에 따라 자동으로 커스텀 구현체를 찾아 주입합니다.

### 3. 실무에서 주의할 점
*   **명명 규칙 준수:** Spring Data JPA가 커스텀 구현체를 자동으로 인식하려면, 커스텀 인터페이스와 구현체의 명명 규칙(`[엔티티명]RepositoryCustom`, `[엔티티명]RepositoryImpl`)을 정확히 지켜야 합니다.
*   **트랜잭션 관리:** 커스텀 구현체 내에서 데이터 변경 로직을 수행할 경우, `@Transactional` 어노테이션을 적절히 사용하여 트랜잭션을 관리해야 합니다. 일반적으로 서비스 계층에서 트랜잭션을 관리하지만, 복잡한 데이터 접근 로직의 경우 리포지토리 계층에서도 필요할 수 있습니다.
*   **쿼리 복잡도:** 단순한 쿼리는 Spring Data JPA의 쿼리 메서드나 `@Query` 어노테이션을 활용하는 것이 더 간결하고 유지보수하기 좋습니다. 커스텀 리포지토리는 정말 복잡하거나 동적인 쿼리, 또는 특정 라이브러리(예: Querydsl)를 활용해야 할 때 사용하는 것이 바람직합니다.
*   **테스트 용이성:** 커스텀 로직은 단위 테스트가 용이하도록 설계하는 것이 중요합니다. `EntityManager`를 직접 사용하는 경우, 테스트 환경에서 이를 모킹하거나 임베디드 데이터베이스를 활용하는 방법을 고려해야 합니다.
