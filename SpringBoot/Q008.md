## Spring IoC 컨테이너와 DI 설명 (모범 답안)

### 1. IoC(Inversion of Control) 컨테이너와 DI(Dependency Injection) 설명

**IoC(Inversion of Control) 컨테이너**는 객체의 생성, 생명 주기 관리, 의존성 주입 등을 개발자가 직접 제어하는 대신, 프레임워크(Spring)가 대신 제어하는 디자인 원칙인 "제어의 역전"을 구현한 핵심 컴포넌트입니다. 개발자는 객체 간의 관계를 정의만 하고, 실제 객체 생성 및 연결은 IoC 컨테이너가 담당합니다.

**DI(Dependency Injection)**는 IoC의 구체적인 구현 패턴 중 하나로, 객체가 직접 의존하는 객체를 생성하거나 찾아오는 대신, 외부(IoC 컨테이너)에서 의존 객체를 주입받는 방식입니다. 즉, "네가 필요한 객체는 내가 줄게"라는 개념입니다.

### 2. DI가 객체 간의 결합도를 낮추고 테스트 용이성을 높이는 방법 (구체적인 예시)

**결합도 감소:**
DI를 사용하지 않으면, `UserService`가 `UserRepository`를 직접 생성하거나 특정 구현체에 강하게 의존하게 됩니다.

```java
// DI 미적용 (높은 결합도)
public class UserService {
    private UserRepository userRepository = new JpaUserRepository(); // 특정 구현체에 직접 의존
    // ...
}
```

DI를 적용하면, `UserService`는 `UserRepository` 인터페이스에만 의존하고, 실제 어떤 구현체(`JpaUserRepository`, `JdbcUserRepository` 등)가 주입될지는 외부(IoC 컨테이너)에서 결정합니다.

```java
// DI 적용 (낮은 결합도)
public interface UserRepository {
    void save(User user);
}

public class JpaUserRepository implements UserRepository {
    // ... JPA 관련 로직
}

public class UserService {
    private final UserRepository userRepository; // 인터페이스에 의존

    public UserService(UserRepository userRepository) { // 생성자를 통해 주입받음
        this.userRepository = userRepository;
    }
    // ...
}
```
이처럼 `UserService`는 `UserRepository`의 구체적인 구현체에 대해 알 필요가 없으므로, `UserRepository`의 구현이 변경되더라도 `UserService` 코드를 수정할 필요가 없어 결합도가 낮아집니다.

**테스트 용이성 증가:**
결합도가 낮아지면 테스트 용이성이 자연스럽게 높아집니다. 위 예시에서 `UserService`를 테스트할 때, 실제 데이터베이스에 접근하는 `JpaUserRepository` 대신, 테스트 목적으로 가짜 `UserRepository` 구현체(Mock 객체)를 쉽게 주입할 수 있습니다.

```java
// UserService 테스트 코드
import org.junit.jupiter.api.Test;
import static org.mockito.Mockito.*;

public class UserServiceTest {

    @Test
    void testSaveUser() {
        // Mock UserRepository 생성
        UserRepository mockUserRepository = mock(UserRepository.class);
        UserService userService = new UserService(mockUserRepository);

        User user = new User("testUser");
        userService.saveUser(user);

        // mockUserRepository의 save 메서드가 호출되었는지 검증
        verify(mockUserRepository, times(1)).save(user);
    }
}
```
`UserService`는 `UserRepository` 인터페이스에만 의존하므로, 테스트 시에는 `mock(UserRepository.class)`를 통해 가짜 `UserRepository`를 생성하여 주입할 수 있습니다. 이를 통해 실제 데이터베이스 연결 없이 `UserService`의 비즈니스 로직만을 독립적으로 테스트할 수 있어 테스트 작성이 훨씬 용이해집니다.

### 3. Spring Boot 환경에서 `@Autowired`와 생성자 주입 방식의 차이점 및 권장되는 주입 방식

**`@Autowired` (필드 주입 또는 세터 주입):**
*   **필드 주입:** 필드에 `@Autowired`를 붙여 직접 주입하는 방식입니다.
    ```java
    @Service
    public class MyService {
        @Autowired
        private MyRepository myRepository; // 필드 주입
        // ...
    }
    ```
*   **세터 주입:** 세터 메서드에 `@Autowired`를 붙여 주입하는 방식입니다.
    ```java
    @Service
    public class MyService {
        private MyRepository myRepository;

        @Autowired
        public void setMyRepository(MyRepository myRepository) { // 세터 주입
            this.myRepository = myRepository;
        }
        // ...
    }
    ```
*   **장점:** 코드가 간결해 보입니다.
*   **단점:**
    *   **단위 테스트 어려움:** `@Autowired`는 Spring 컨테이너가 관리할 때만 동작하므로, 순수 Java 환경에서 단위 테스트 시 의존 객체를 직접 주입하기 어렵습니다. (리플렉션 등을 사용해야 함)
    *   **불변성 보장 어려움:** 객체 생성 후 필드가 변경될 수 있어 불변성을 보장하기 어렵습니다.
    *   **순환 참조 문제:** 순환 참조 발생 시 애플리케이션 구동 시점에 오류를 발견하기 어렵습니다.
    *   **단일 책임 원칙 위배 가능성:** 필드가 많아지면 의존성이 많다는 것을 한눈에 파악하기 어려워, 클래스가 너무 많은 책임을 가질 가능성이 있습니다.

**생성자 주입:**
*   클래스의 생성자를 통해 의존 객체를 주입받는 방식입니다. Spring 4.3부터는 단일 생성자의 경우 `@Autowired`를 생략해도 자동으로 주입됩니다.
    ```java
    @Service
    public class MyService {
        private final MyRepository myRepository; // final 키워드 사용

        // @Autowired 생략 가능 (단일 생성자의 경우)
        public MyService(MyRepository myRepository) { // 생성자 주입
            this.myRepository = myRepository;
        }
        // ...
    }
    ```
*   **장점:**
    *   **불변성 보장:** `final` 키워드를 사용하여 객체 생성 후 의존 객체가 변경되지 않음을 보장할 수 있습니다.
    *   **테스트 용이성:** 순수 Java 코드에서도 생성자를 통해 의존 객체를 쉽게 주입할 수 있어 단위 테스트가 용이합니다.
    *   **순환 참조 방지:** 순환 참조 발생 시 애플리케이션 구동 시점에 `BeanCurrentlyInCreationException` 등의 오류를 발생시켜 문제를 조기에 발견할 수 있습니다.
    *   **의존성 명확성:** 생성자의 파라미터를 통해 해당 클래스가 어떤 의존성을 가지는지 명확하게 알 수 있습니다. 의존성이 많아지면 생성자 파라미터가 많아져 리팩토링의 필요성을 인지하기 쉽습니다.
    *   **DI 컨테이너 없이도 객체 생성 가능:** Spring 컨테이너 없이도 일반 Java 객체처럼 생성자를 통해 객체를 생성하고 테스트할 수 있습니다.

**권장되는 주입 방식:**

**생성자 주입**이 가장 권장되는 방식입니다. 특히 `final` 키워드와 함께 사용하여 불변성을 확보하고, 테스트 용이성을 높이며, 순환 참조 문제를 조기에 발견할 수 있다는 강력한 장점들이 있기 때문입니다.

Lombok의 `@RequiredArgsConstructor` 어노테이션을 사용하면 `final` 필드에 대한 생성자를 자동으로 생성해주어 코드를 더욱 간결하게 유지하면서 생성자 주입의 장점을 누릴 수 있습니다.

```java
import lombok.RequiredArgsConstructor;
import org.springframework.stereotype.Service;

@Service
@RequiredArgsConstructor // final 필드에 대한 생성자를 자동으로 생성
public class MyService {
    private final MyRepository myRepository;
    // ...
}
```
