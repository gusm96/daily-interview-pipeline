## Spring WebFlux 면접 질문 모범 답안

### 1. Spring MVC와 Spring WebFlux의 아키텍처적 차이점

**Spring MVC**는 서블릿(Servlet) 기반의 동기 블로킹(Blocking) 아키텍처입니다. 요청당 하나의 스레드가 할당되어 작업이 완료될 때까지 해당 스레드를 점유합니다. 이는 I/O 작업(데이터베이스 접근, 외부 API 호출 등)이 많은 환경에서 스레드 고갈 및 컨텍스트 스위칭 오버헤드로 인해 성능 저하를 야기할 수 있습니다.

**Spring WebFlux**는 비동기 논블로킹(Non-blocking) 아키텍처로, 리액터(Reactor)와 같은 반응형 스트림(Reactive Streams) 라이브러리를 기반으로 합니다. 적은 수의 스레드로 많은 요청을 효율적으로 처리하며, I/O 작업 시 스레드가 블로킹되지 않고 다른 작업을 수행할 수 있도록 합니다. 이는 이벤트 루프(Event Loop) 모델과 유사하게 동작하여, 작업 완료 시 콜백(Callback) 또는 반응형 스트림을 통해 결과를 전달합니다.

**주요 차이점 요약:**

| 특징         | Spring MVC (서블릿 기반)                 | Spring WebFlux (리액티브 기반)             |
| :----------- | :--------------------------------------- | :----------------------------------------- |
| **동작 방식**  | 동기 블로킹                              | 비동기 논블로킹                            |
| **스레드 모델** | 요청당 스레드 할당 (1:1)                 | 적은 수의 스레드로 다수 요청 처리 (N:M)    |
| **I/O 처리**   | I/O 작업 시 스레드 블로킹                | I/O 작업 시 스레드 블로킹 없음            |
| **백프레셔**   | 지원 안 함                               | 지원 (생산자-소비자 속도 조절)             |
| **주요 라이브러리** | Servlet API, Tomcat, Jetty             | Reactor, Netty, Undertow                   |

### 2. Reactor 라이브러리의 `Mono`와 `Flux` 및 반응형 프로그래밍의 기본 개념과 장점

**Reactor**는 JVM에서 반응형 프로그래밍을 위한 라이브러리로, Reactive Streams 사양을 구현합니다. 핵심 요소는 `Mono`와 `Flux`입니다.

*   **`Mono`**: 0개 또는 1개의 데이터를 비동기적으로 방출(emit)하는 반응형 스트림입니다. 단일 결과값(예: 사용자 정보 조회)을 처리할 때 사용됩니다.
*   **`Flux`**: 0개에서 N개의 데이터를 비동기적으로 방출하는 반응형 스트림입니다. 여러 결과값(예: 게시글 목록 조회)을 처리할 때 사용됩니다.

**반응형 프로그래밍의 기본 개념:**

데이터 스트림과 변경 사항의 전파에 중점을 둔 프로그래밍 패러다임입니다. 데이터는 스트림을 통해 흐르고, 이 스트림에 대한 연산(map, filter, flatMap 등)을 체인 형태로 연결하여 데이터를 변환하고 처리합니다. 데이터가 준비될 때까지 기다리지 않고, 데이터가 도착하면 미리 정의된 로직을 실행하는 방식으로 동작합니다.

**반응형 프로그래밍의 장점:**

*   **적은 스레드로 높은 동시성 처리**: I/O 작업 시 스레드가 블로킹되지 않고 다른 요청을 처리할 수 있으므로, 적은 수의 스레드로 훨씬 많은 동시 요청을 처리할 수 있습니다. 이는 스레드 생성 및 컨텍스트 스위칭 오버헤드를 줄여줍니다.
*   **자원 효율성**: 스레드 풀의 크기를 작게 유지하여 메모리 사용량을 줄이고 CPU 활용도를 높일 수 있습니다.
*   **백프레셔(Backpressure) 지원**: 생산자가 소비자의 처리 속도보다 빠르게 데이터를 생산할 때, 소비자가 생산자에게 데이터 전송 속도를 조절하도록 요청할 수 있는 메커니즘입니다. 이는 시스템 과부하를 방지하고 안정성을 높입니다.
*   **코드의 간결성 및 가독성**: 비동기 코드를 콜백 지옥(Callback Hell) 없이 선언적이고 함수형 스타일로 작성할 수 있어 코드가 간결하고 이해하기 쉽습니다.
*   **장애 격리 및 복원력**: 스트림 내에서 발생하는 오류를 효과적으로 처리하고 복구 로직을 적용하기 용이합니다.

### 3. WebFlux가 대용량 트래픽 환경에서 제공하는 이점

Spring WebFlux는 대용량 트래픽 환경에서 다음과 같은 이점을 제공합니다.

*   **높은 확장성(Scalability)**: 적은 수의 스레드로 많은 동시 요청을 처리할 수 있어, 서버 자원(CPU, 메모리)을 효율적으로 사용하여 더 많은 트래픽을 감당할 수 있습니다. 이는 수평적 확장을 용이하게 합니다.
*   **낮은 지연 시간(Low Latency)**: I/O 작업으로 인한 스레드 블로킹이 없어, 요청 처리 지연 시간이 줄어들고 응답성이 향상됩니다. 특히 마이크로서비스 아키텍처에서 여러 서비스 간의 통신이 빈번할 때 전체 시스템의 지연 시간을 줄이는 데 기여합니다.
*   **자원 효율성**: 스레드 풀 크기를 작게 유지하여 메모리 사용량을 줄이고, CPU를 I/O 대기 없이 실제 작업에 더 많이 활용할 수 있습니다. 이는 클라우드 환경에서 운영 비용 절감으로 이어질 수 있습니다.
*   **안정적인 성능**: 백프레셔 메커니즘을 통해 시스템 과부하를 방지하고, 갑작스러운 트래픽 증가에도 안정적인 성능을 유지할 수 있도록 돕습니다.
*   **마이크로서비스 및 이벤트 기반 아키텍처에 적합**: 비동기 논블로킹 특성상 마이크로서비스 간의 효율적인 통신(예: WebClient를 이용한 비동기 HTTP 호출)과 이벤트 기반 시스템 구축에 매우 유리합니다.

결론적으로 Spring WebFlux는 높은 동시성, 낮은 지연 시간, 효율적인 자원 사용이 요구되는 대용량 트래픽 환경에서 기존 Spring MVC보다 뛰어난 성능과 확장성을 제공하는 현대적인 웹 프레임워크입니다.
