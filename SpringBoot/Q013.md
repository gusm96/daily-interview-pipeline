## Spring Data JPA N+1 문제 해결 전략 (모범 답안)

### 1. N+1 문제의 원인

Spring Data JPA에서 N+1 문제는 **연관 관계를 가진 엔티티를 조회할 때, 초기 쿼리 한 번(1) 이후 연관된 엔티티의 개수(N)만큼 추가 쿼리가 발생하는 현상**을 의미합니다. 이는 주로 **지연 로딩(Lazy Loading)** 전략과 함께 발생합니다.

**예시:** `Order` 엔티티가 `OrderItem` 엔티티 리스트를 가지고 있다고 가정해봅시다.

```java
@Entity
public class Order {
    @Id @GeneratedValue
    private Long id;

    @OneToMany(mappedBy = "order", fetch = FetchType.LAZY) // 지연 로딩
    private List<OrderItem> orderItems = new ArrayList<>();

    // ...
}

@Entity
public class OrderItem {
    @Id @GeneratedValue
    private Long id;

    @ManyToOne(fetch = FetchType.LAZY) // 지연 로딩
    @JoinColumn(name = "order_id")
    private Order order;

    // ...
}
```

`Order` 목록을 조회한 후 각 `Order`의 `orderItems`에 접근할 때 N+1 문제가 발생합니다.

```java
List<Order> orders = orderRepository.findAll(); // 1번 쿼리: 모든 Order 조회

for (Order order : orders) {
    // order.getOrderItems() 호출 시, 각 Order마다 OrderItem을 조회하는 쿼리(N번) 발생
    System.out.println("Order ID: " + order.getId() + ", Items: " + order.getOrderItems().size());
}
```

이 경우, `Order`가 100개라면 `Order` 조회 쿼리 1번과 `OrderItem` 조회 쿼리 100번, 총 101번의 쿼리가 실행되어 성능 저하를 초래합니다.

### 2. N+1 문제 해결 전략

N+1 문제를 해결하기 위한 다양한 전략들은 크게 **즉시 로딩(Eager Loading)으로 변경**하거나, **쿼리 최적화**하는 방식으로 나눌 수 있습니다.

#### 2.1. Fetch Join

**설명:** JPQL(Java Persistence Query Language) 또는 Criteria API를 사용하여 연관된 엔티티를 **즉시 로딩(Eager Loading)** 방식으로 한 번의 쿼리로 함께 조회하는 전략입니다. `JOIN FETCH` 키워드를 사용합니다.

**예시:**

```java
// OrderRepository.java
public interface OrderRepository extends JpaRepository<Order, Long> {
    @Query("SELECT o FROM Order o JOIN FETCH o.orderItems")
    List<Order> findAllWithOrderItemsFetchJoin();
}
```

**장점:**
*   **가장 효과적인 해결책:** 대부분의 N+1 문제를 한 번의 쿼리로 해결하여 성능을 크게 향상시킵니다.
*   **간단한 문법:** JPQL을 통해 직관적으로 사용할 수 있습니다.

**단점:**
*   **카테시안 곱(Cartesian Product) 발생 가능성:** `@OneToMany` 관계에서 Fetch Join을 사용하면 결과 집합의 행 수가 증가하여 메모리 사용량이 늘어날 수 있습니다. (예: `Order` 1개에 `OrderItem` 3개가 있으면 `Order`가 3번 중복되어 조회됨)
*   **페이징 처리의 어려움:** Fetch Join과 함께 `LIMIT` (페이징)을 사용하면 데이터베이스 레벨에서 페이징이 아닌 애플리케이션 레벨에서 페이징이 되어 성능 문제가 발생할 수 있습니다. (Hibernate 6부터는 개선되었지만 여전히 주의 필요)
*   **재사용성 제한:** 특정 쿼리에 종속적이므로 다른 조회에서는 다시 Fetch Join을 명시해야 합니다.

**적합한 상황:**
*   대부분의 경우 N+1 문제를 해결하는 데 가장 먼저 고려해야 할 전략입니다.
*   연관된 엔티티의 수가 많지 않고, 페이징이 필요 없는 단일 조회 또는 전체 조회 시 효과적입니다.
*   `@ManyToOne`, `@OneToOne` 관계에서 특히 강력합니다.

#### 2.2. `@EntityGraph`

**설명:** JPA 2.1부터 도입된 기능으로, 엔티티의 특정 속성(연관 관계)을 즉시 로딩하도록 힌트를 제공하는 전략입니다. `@NamedEntityGraph` 또는 `@EntityGraph` 어노테이션을 사용하여 정의합니다.

**예시:**

```java
// Order.java
@Entity
@NamedEntityGraph(name = "Order.withOrderItems", attributeNodes = @NamedAttributeNode("orderItems"))
public class Order {
    // ...
}

// OrderRepository.java
public interface OrderRepository extends JpaRepository<Order, Long> {
    @EntityGraph(value = "Order.withOrderItems", type = EntityGraph.EntityGraphType.LOAD) // LOAD: 기본적으로 정의된 그래프를 사용하고, 추가로 필요한 필드는 지연 로딩
    //@EntityGraph(attributePaths = {"orderItems"}, type = EntityGraph.EntityGraphType.FETCH) // FETCH: 정의된 그래프만 즉시 로딩하고, 나머지는 지연 로딩
    List<Order> findAllWithOrderItemsEntityGraph();
}
```

**장점:**
*   **유연성:** 여러 엔티티 그래프를 정의하여 다양한 조회 요구사항에 맞춰 재사용할 수 있습니다.
*   **쿼리 재사용성:** Repository 메서드에 어노테이션만 추가하면 되므로, JPQL을 직접 작성하는 것보다 간편합니다.
*   **페이징 처리 가능:** Fetch Join과 달리 `@EntityGraph`는 페이징과 함께 사용해도 비교적 안전합니다. (단, `type = EntityGraph.EntityGraphType.FETCH` 사용 시 주의)

**단점:**
*   **카테시안 곱 발생 가능성:** Fetch Join과 마찬가지로 `@OneToMany` 관계에서 발생할 수 있습니다.
*   **복잡한 그래프 정의:** 여러 계층의 연관 관계를 즉시 로딩하려면 `@NamedAttributeNode`를 중첩하여 정의해야 하므로 복잡해질 수 있습니다.
*   **JPQL/Criteria API보다 제한적:** 복잡한 조건이나 서브 쿼리 등은 `@EntityGraph`로 표현하기 어렵습니다.

**적합한 상황:**
*   다양한 조회 시나리오에서 특정 연관 관계를 즉시 로딩해야 할 때.
*   Fetch Join의 카테시안 곱 문제나 페이징 문제로 인해 대안이 필요할 때.
*   Repository 메서드 레벨에서 선언적으로 N+1 문제를 해결하고 싶을 때.

#### 2.3. Batch Size 설정 (Hibernate `default_batch_fetch_size`)

**설명:** JPA 구현체(Hibernate)의 설정으로, 지연 로딩되는 연관 엔티티들을 한 번에 여러 개씩 묶어서 조회하도록 하는 전략입니다. `application.properties` 또는 `application.yml`에 `spring.jpa.properties.hibernate.default_batch_fetch_size` 속성을 설정합니다.

**예시:**

```yaml
# application.yml
spring:
  jpa:
    properties:
      hibernate:
        default_batch_fetch_size: 100 # 100개씩 묶어서 조회
```

**동작 방식:**
`Order` 목록을 조회한 후 `order.getOrderItems()`에 접근할 때, Hibernate는 `orderItems`를 조회하는 쿼리를 즉시 실행하지 않고, `default_batch_fetch_size`에 설정된 개수만큼의 `Order` ID를 모아서 한 번의 `IN` 쿼리로 `OrderItem`들을 조회합니다.

```sql
-- 첫 번째 쿼리 (Order 조회)
SELECT * FROM orders;

-- 두 번째 쿼리 (OrderItem 조회 - batch size 100 가정)
SELECT * FROM order_items WHERE order_id IN (1, 2, ..., 100);
SELECT * FROM order_items WHERE order_id IN (101, 102, ..., 200);
-- ...
```

**장점:**
*   **코드 변경 없음:** 기존 코드에 영향을 주지 않고 설정만으로 N+1 문제를 완화할 수 있습니다.
*   **유연한 적용:** 모든 지연 로딩 연관 관계에 일괄적으로 적용됩니다.
*   **페이징과 호환:** 페이징 처리와 함께 사용해도 문제가 없습니다.
*   **카테시안 곱 없음:** Fetch Join과 달리 카테시안 곱이 발생하지 않습니다.

**단점:**
*   **완벽한 해결책은 아님:** 여전히 N번의 쿼리가 아닌 N/BatchSize 번의 쿼리가 발생하므로, 쿼리 수가 줄어들 뿐 완전히 한 번의 쿼리로 해결되지는 않습니다.
*   **적절한 Batch Size 설정의 중요성:** 너무 작으면 쿼리 수가 많아지고, 너무 크면 `IN` 절의 길이가 길어져 데이터베이스 부하가 커질 수 있습니다.
*   **즉시 로딩이 필요한 경우 부적합:** 특정 연관 관계를 항상 즉시 로딩해야 하는 경우에는 Fetch Join이나 `@EntityGraph`가 더 적합합니다.

**적합한 상황:**
*   코드 변경 없이 N+1 문제를 완화하고 싶을 때.
*   페이징 처리와 함께 N+1 문제를 해결해야 할 때.
*   `@OneToMany` 관계에서 카테시안 곱을 피하고 싶을 때.
*   지연 로딩이 기본 전략이지만, 특정 시점에 여러 연관 엔티티를 효율적으로 로딩해야 할 때.

### 3. 각 전략 비교 및 선택 가이드

| 전략             | 쿼리 수 (N+1 문제 해결 정도) | 카테시안 곱 | 페이징 호환성 | 코드 변경 | 재사용성 | 장점                                                              | 단점                                                              |
| :--------------- | :--------------------------- | :---------- | :------------ | :-------- | :------- | :---------------------------------------------------------------- | :---------------------------------------------------------------- |
| **Fetch Join**   | 1 (가장 효과적)              | 발생 가능   | 어려움        | 높음      | 낮음     | 가장 강력한 성능 향상, 직관적                                     | 카테시안 곱, 페이징 문제, 특정 쿼리에 종속적                      |
| **`@EntityGraph`** | 1 (효과적)                   | 발생 가능   | 좋음          | 중간      | 높음     | 유연한 그래프 정의, 페이징 호환, 선언적                           | 카테시안 곱, 복잡한 그래프 정의, JPQL보다 제한적                  |
| **Batch Size**   | N/BatchSize (완화)           | 없음        | 좋음          | 낮음      | 높음     | 코드 변경 없음, 페이징 호환, 카테시안 곱 없음                     | 완벽한 해결책 아님, 적절한 Batch Size 설정 필요                   |

**선택 가이드:**

1.  **가장 먼저 Fetch Join을 고려합니다.** 대부분의 N+1 문제를 가장 효과적으로 해결할 수 있습니다.
2.  **Fetch Join 사용 시 카테시안 곱이나 페이징 문제가 발생한다면 `@EntityGraph`를 고려합니다.** `@EntityGraph`는 Fetch Join의 단점을 보완하면서도 강력한 성능 향상을 제공합니다.
3.  **코드 변경 없이 전역적으로 N+1 문제를 완화하고 싶거나, 페이징과 함께 사용해야 하며, 완벽한 1번 쿼리가 필수는 아닐 때 Batch Size 설정을 고려합니다.** 이는 Fetch Join이나 `@EntityGraph`와 함께 보조적인 전략으로 사용될 수도 있습니다.
4.  **복잡한 조회 요구사항이나 DTO 변환이 필요한 경우, QueryDSL과 같은 라이브러리를 사용하여 DTO Projection을 직접 구현하는 것도 좋은 방법입니다.** 이는 N+1 문제를 넘어 필요한 데이터만 조회하여 네트워크 부하까지 줄일 수 있습니다.

각 전략의 장단점을 이해하고, 애플리케이션의 특정 요구사항과 성능 목표에 맞춰 적절한 전략을 선택하는 것이 중요합니다.
