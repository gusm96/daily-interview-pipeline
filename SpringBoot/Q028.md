## Spring Security 인증(Authentication) 및 인가(Authorization) 과정

Spring Security는 서블릿 필터 체인을 기반으로 동작하며, 크게 인증과 인가 두 가지 과정을 통해 애플리케이션 보안을 강화합니다.

### 1. 인증(Authentication) 과정

사용자가 누구인지 확인하는 과정입니다.

*   **`SecurityContextHolder`**: Spring Security의 핵심으로, 현재 스레드와 관련된 `SecurityContext`를 저장합니다. `SecurityContext`는 현재 인증된 사용자(`Authentication` 객체) 정보를 담고 있습니다. 기본적으로 `ThreadLocal`을 사용하여 스레드마다 독립적인 보안 컨텍스트를 유지합니다.
*   **`Authentication`**: 현재 사용자의 인증 정보를 나타내는 객체입니다. `Principal`(사용자 식별 정보), `Credentials`(비밀번호 등), `Authorities`(권한 목록)를 포함합니다. 인증 전에는 `isAuthenticated()`가 `false`이고, 인증 후에는 `true`가 됩니다.
*   **`AuthenticationManager`**: `Authentication` 객체를 받아 인증을 시도하고, 성공하면 완전히 인증된 `Authentication` 객체를 반환합니다. 실패하면 `AuthenticationException`을 발생시킵니다.
    *   **`ProviderManager`**: `AuthenticationManager`의 가장 일반적인 구현체로, 여러 `AuthenticationProvider`를 관리합니다.
*   **`AuthenticationProvider`**: 특정 유형의 `Authentication`을 처리하는 실제 인증 로직을 수행합니다. 예를 들어, `DaoAuthenticationProvider`는 사용자 이름/비밀번호를 기반으로 데이터베이스에서 사용자 정보를 조회하여 인증합니다.

**동작 흐름:**

1.  사용자가 로그인 요청을 보내면, `UsernamePasswordAuthenticationFilter`와 같은 인증 필터가 요청을 가로챕니다.
2.  필터는 요청에서 사용자 이름과 비밀번호를 추출하여 `UsernamePasswordAuthenticationToken`과 같은 `Authentication` 객체를 생성합니다. (이때는 아직 인증되지 않은 상태)
3.  생성된 `Authentication` 객체를 `AuthenticationManager`에게 전달하여 인증을 요청합니다.
4.  `AuthenticationManager` (주로 `ProviderManager`)는 등록된 `AuthenticationProvider`들을 순회하며 해당 `Authentication` 객체를 처리할 수 있는 프로바이더를 찾습니다.
5.  적절한 `AuthenticationProvider`가 사용자 이름과 비밀번호를 검증하고, 성공하면 완전히 인증된 `Authentication` 객체를 반환합니다.
6.  `AuthenticationManager`는 이 인증된 `Authentication` 객체를 `SecurityContextHolder`에 저장합니다. 이제 애플리케이션의 어느 곳에서든 `SecurityContextHolder.getContext().getAuthentication()`을 통해 현재 사용자 정보를 얻을 수 있습니다.

### 2. 인가(Authorization) 과정

인증된 사용자가 특정 리소스(URL, 메서드 등)에 접근할 권한이 있는지 확인하는 과정입니다.

*   **`AccessDecisionManager`**: `Authentication` 객체(인증된 사용자 정보), `Object` (보호되는 리소스), `Collection<ConfigAttribute>` (리소스에 필요한 권한 정보)를 받아 접근 허용 여부를 결정합니다.
    *   **`AffirmativeBased`**: 하나라도 접근을 허용하는 `AccessDecisionVoter`가 있으면 접근을 허용합니다.
    *   **`ConsensusBased`**: 과반수의 `AccessDecisionVoter`가 접근을 허용하면 접근을 허용합니다.
    *   **`UnanimousBased`**: 모든 `AccessDecisionVoter`가 접근을 허용해야만 접근을 허용합니다.
*   **`AccessDecisionVoter`**: `AccessDecisionManager`에 의해 사용되며, 실제 접근 결정 투표를 수행합니다. `ACCESS_GRANTED`, `ACCESS_DENIED`, `ACCESS_ABSTAIN` 중 하나를 반환합니다. 예를 들어, `RoleVoter`는 사용자의 역할과 리소스에 필요한 역할을 비교하여 투표합니다.
*   **`FilterSecurityInterceptor`**: URL 기반의 인가를 처리하는 핵심 필터입니다. 요청된 URL에 접근하기 전에 `AccessDecisionManager`를 호출하여 인가 여부를 확인합니다.

**동작 흐름:**

1.  인증이 완료된 후, 사용자가 특정 URL에 접근을 시도하면 `FilterSecurityInterceptor`가 요청을 가로챕니다.
2.  `FilterSecurityInterceptor`는 해당 URL에 설정된 `ConfigAttribute` (예: `hasRole('ADMIN')`)를 가져옵니다.
3.  `FilterSecurityInterceptor`는 `SecurityContextHolder`에서 현재 인증된 `Authentication` 객체를 가져옵니다.
4.  `FilterSecurityInterceptor`는 `Authentication` 객체와 `ConfigAttribute`를 `AccessDecisionManager`에게 전달하여 인가 결정을 요청합니다.
5.  `AccessDecisionManager`는 등록된 `AccessDecisionVoter`들에게 투표를 요청합니다.
6.  `AccessDecisionVoter`들은 `Authentication` 객체의 권한과 `ConfigAttribute`를 비교하여 접근 허용 여부를 투표합니다.
7.  `AccessDecisionManager`는 투표 결과를 종합하여 최종적으로 접근을 허용할지 거부할지 결정합니다.
8.  접근이 허용되면 요청은 계속 진행되고, 거부되면 `AccessDeniedException`이 발생합니다.

## JWT(JSON Web Token) 기반 인증 시스템 통합 전략 및 주의사항

JWT는 클라이언트와 서버 간에 정보를 안전하게 교환하기 위한 간결하고 자체 포함적인 방법입니다. Spring Security와 통합하여 Stateless한 인증 시스템을 구축할 수 있습니다.

### 구현 전략

1.  **JWT 생성 및 발급**:
    *   사용자가 로그인 요청(ID/PW)을 보내면, `AuthenticationManager`를 통해 인증을 수행합니다.
    *   인증에 성공하면, `Authentication` 객체에서 사용자 정보를 추출하여 JWT를 생성합니다. (예: `jjwt` 라이브러리 사용)
    *   생성된 JWT를 클라이언트에게 응답으로 보냅니다. (일반적으로 HTTP Header의 `Authorization` 필드에 `Bearer` 토큰 형태로)
2.  **JWT 검증 필터 구현**:
    *   `OncePerRequestFilter`를 상속받아 `JwtAuthenticationFilter`와 같은 커스텀 필터를 구현합니다.
    *   이 필터는 `UsernamePasswordAuthenticationFilter`보다 **앞에** 위치하도록 설정합니다.
    *   모든 요청에 대해 `Authorization` 헤더에서 JWT를 추출합니다.
    *   추출된 JWT의 유효성을 검증합니다. (서명 확인, 만료일 확인 등)
    *   JWT가 유효하면, JWT에서 사용자 정보(Principal, Authorities)를 파싱하여 `UsernamePasswordAuthenticationToken` 또는 커스텀 `Authentication` 객체를 생성합니다.
    *   생성된 `Authentication` 객체를 `SecurityContextHolder.getContext().setAuthentication()`을 통해 `SecurityContext`에 저장합니다.
    *   `chain.doFilter(request, response)`를 호출하여 다음 필터로 요청을 전달합니다.
3.  **Spring Security 설정**:
    *   `WebSecurityConfigurerAdapter`를 상속받아 `configure(HttpSecurity http)` 메서드를 오버라이드합니다.
    *   **세션 비활성화**: `http.sessionManagement().sessionCreationPolicy(SessionCreationPolicy.STATELESS)`를 설정하여 세션을 사용하지 않도록 합니다. JWT는 자체 포함적이므로 서버에 세션 정보를 저장할 필요가 없습니다.
    *   **CSRF 비활성화**: `http.csrf().disable()`를 설정합니다. 세션을 사용하지 않는 REST API에서는 CSRF 공격에 대한 위험이 낮으므로 비활성화하는 경우가 많습니다.
    *   **인증 필터 등록**: `http.addFilterBefore(jwtAuthenticationFilter, UsernamePasswordAuthenticationFilter.class)`를 사용하여 구현한 JWT 검증 필터를 등록합니다.
    *   **인가 규칙 설정**: `http.authorizeRequests()`를 사용하여 특정 URL 패턴에 대한 접근 권한을 설정합니다. (예: `/api/public/**`은 허용, `/api/admin/**`은 `hasRole('ADMIN')` 필요)
    *   **예외 처리**: `http.exceptionHandling().authenticationEntryPoint(jwtAuthenticationEntryPoint)`를 설정하여 인증되지 않은 요청에 대한 응답을 커스터마이징할 수 있습니다.

### 주의해야 할 점

1.  **JWT 보안**:
    *   **Secret Key 관리**: JWT 서명에 사용되는 Secret Key는 외부에 노출되지 않도록 안전하게 관리해야 합니다. 환경 변수, 키 저장소 등을 활용합니다.
    *   **HTTPS 사용**: JWT는 암호화되지 않으므로, 전송 중 가로채기(Man-in-the-Middle) 공격을 방지하기 위해 반드시 HTTPS를 사용해야 합니다.
    *   **토큰 탈취**: 탈취된 JWT는 만료될 때까지 유효하므로, 토큰의 유효 기간을 적절히 설정하고, 필요시 토큰 무효화(블랙리스트) 메커니즘을 고려해야 합니다.
2.  **토큰 만료 및 갱신**:
    *   **짧은 만료 시간**: 보안을 위해 JWT의 만료 시간을 짧게 설정하는 것이 좋습니다.
    *   **Refresh Token 도입**: 만료 시간이 짧은 Access Token과 만료 시간이 긴 Refresh Token을 함께 사용하여 사용자 경험을 저해하지 않으면서 보안을 강화할 수 있습니다. Access Token이 만료되면 Refresh Token을 사용하여 새로운 Access Token을 발급받습니다.
3.  **블랙리스트/무효화**:
    *   JWT는 Stateless하므로 서버에서 직접 무효화하기 어렵습니다. 로그아웃, 비밀번호 변경 등으로 토큰을 무효화해야 할 경우, Redis와 같은 캐시 저장소에 무효화된 토큰(블랙리스트)을 저장하고, 요청 시마다 해당 토큰이 블랙리스트에 있는지 확인하는 로직을 추가해야 합니다.
4.  **정보 과부하**:
    *   JWT Payload에 너무 많은 정보를 담으면 토큰의 크기가 커지고, 매 요청마다 전송되므로 네트워크 오버헤드가 발생할 수 있습니다. 필요한 최소한의 정보만 담는 것이 좋습니다.
5.  **클라이언트 측 저장**:
    *   클라이언트(브라우저)에서 JWT를 저장할 때 `localStorage`보다는 `HttpOnly` 및 `Secure` 플래그가 설정된 `cookie`에 저장하는 것이 XSS(Cross-Site Scripting) 공격으로부터 더 안전합니다.
6.  **예외 처리**:
    *   JWT가 없거나, 유효하지 않거나, 만료되었을 때 적절한 HTTP 상태 코드(예: 401 Unauthorized)와 메시지를 반환하도록 예외 처리를 구현해야 합니다. `AuthenticationEntryPoint`를 활용합니다.
