# [Q003] Spring AOP 프록시 동작과 AspectJ 차이

> 🌱 Spring Boot · 2026-06-19

**Q.** Spring Framework에서 AOP(Aspect-Oriented Programming)는 횡단 관심사(Cross-cutting Concerns)를 모듈화하는 데 사용됩니다. Spring AOP에서 프록시 기반 AOP가 어떻게 동작하는지 설명하고, AspectJ와의 주요 차이점 및 Spring Boot 환경에서 AOP를 활용하는 일반적인 시나리오를 설명해주세요.

## 🧑‍💻 나의 답변

## Spring AOP 모범 답안 (간결한 한국어)

### 1. Spring AOP의 프록시 기반 동작 방식

Spring AOP는 **런타임 프록시 기반**으로 동작합니다. 핵심 원리는 다음과 같습니다.

*   **대상 객체 래핑:** Spring 컨테이너는 AOP 어드바이스(Advice)가 적용될 대상 객체(Target Object)를 직접 수정하지 않습니다. 대신, 런타임에 대상 객체를 감싸는 **프록시 객체(Proxy Object)**를 생성합니다.
*   **프록시의 역할:** 클라이언트가 대상 객체를 호출할 때, 실제로는 이 프록시 객체를 호출하게 됩니다. 프록시는 호출을 가로채서(Intercept) 미리 정의된 어드바이스(예: `@Before`, `@After`, `@Around`)를 실행한 후, 원래 대상 객체의 메서드를 호출합니다.
*   **JDK 동적 프록시 vs. CGLIB:**
  *   **인터페이스 기반:** 대상 객체가 하나 이상의 인터페이스를 구현하는 경우, Spring은 **JDK 동적 프록시**를 사용합니다. 이 프록시는 대상 객체의 인터페이스를 구현하여 생성됩니다.
  *   **클래스 기반:** 대상 객체가 인터페이스를 구현하지 않거나, 클래스 기반 프록시를 명시적으로 설정한 경우, Spring은 **CGLIB(Code Generation Library)**를 사용하여 대상 객체의 서브클래스를 생성하여 프록시로 활용합니다.

### 2. Spring AOP와 AspectJ의 주요 차이점

| 특징         | Spring AOP                                     | AspectJ                                                              |
| :----------- | :--------------------------------------------- | :------------------------------------------------------------------- |
| **구현 방식** | **런타임 프록시 기반** (JDK 동적 프록시 또는 CGLIB) | **컴파일 타임 또는 로드 타임 위빙(Weaving)** (바이트코드 직접 조작) |
| **적용 시점** | **런타임** (Spring 컨테이너가 빈 생성 시)       | **컴파일 타임** (소스 코드 -> 바이트코드), **로드 타임** (JVM 로딩 시) |
| **대상**     | **Spring 관리 빈(Bean)**에 한정                 | **모든 Java 객체** (Spring 관리 여부 무관)                           |
| **기능 범위** | **메서드 실행 조인 포인트**에 주로 한정         | **필드 접근, 객체 생성, 예외 처리 등** 더 넓은 조인 포인트 지원      |
| **복잡성**   | 상대적으로 **간단하고 가볍다**                 | 상대적으로 **강력하지만 복잡하고 학습 곡선이 높다**                  |
| **의존성**   | Spring Core에 내장                               | 별도의 AspectJ 라이브러리 및 컴파일러/로더 필요                      |

**요약:** Spring AOP는 Spring 컨테이너 내에서 가볍게 AOP를 적용하기 위한 반면, AspectJ는 더 강력하고 포괄적인 AOP 솔루션으로, Spring 외부에서도 사용할 수 있습니다.

### 3. Spring Boot 환경에서 AOP 활용 일반적인 시나리오

Spring Boot는 AOP를 쉽게 설정하고 활용할 수 있도록 자동 구성을 제공합니다. 일반적인 시나리오는 다음과 같습니다.

*   **로깅(Logging):** 특정 메서드 실행 전후, 또는 예외 발생 시 자동으로 로그를 기록하여 애플리케이션의 동작을 추적하고 디버깅하는 데 활용됩니다.
  *   **예시:** `@Before` 어드바이스로 메서드 진입 시 로그, `@AfterReturning`으로 메서드 성공 시 결과 로그, `@AfterThrowing`으로 예외 발생 시 에러 로그.
*   **성능 모니터링(Performance Monitoring):** 메서드 실행 시간을 측정하여 병목 현상을 식별하고 성능 최적화에 기여합니다.
  *   **예시:** `@Around` 어드바이스로 메서드 실행 전 `System.nanoTime()`을 기록하고, 실행 후 다시 측정하여 소요 시간 계산.
*   **트랜잭션 관리(Transaction Management):** 선언적 트랜잭션 관리는 Spring AOP의 대표적인 활용 사례입니다. `@Transactional` 어노테이션을 통해 메서드 단위로 트랜잭션을 적용합니다. (내부적으로 AOP 프록시가 트랜잭션 시작/커밋/롤백 로직을 처리)
*   **보안(Security):** 특정 메서드에 접근하기 전에 사용자 인증 및 권한 검사를 수행하여 보안 정책을 중앙 집중식으로 관리합니다.
  *   **예시:** `@PreAuthorize`와 같은 Spring Security 어노테이션은 AOP를 활용하여 메서드 호출 전 권한 검사를 수행합니다.
*   **캐싱(Caching):** 메서드 호출 결과를 캐시하여 동일한 요청에 대해 불필요한 연산을 줄이고 응답 속도를 향상시킵니다.
  *   **예시:** `@Cacheable` 어노테이션을 통해 메서드 실행 결과를 캐시하고, 다음 호출 시 캐시된 데이터를 반환합니다.
*   **예외 처리(Exception Handling):** 특정 타입의 예외가 발생했을 때 공통적인 예외 처리 로직(예: 에러 로깅, 사용자 친화적인 메시지 반환)을 적용합니다.
  *   **예시:** `@AfterThrowing` 어드바이스로 특정 예외 발생 시 공통 예외 핸들러 호출.

이러한 시나리오들은 횡단 관심사를 비즈니스 로직과 분리하여 코드의 모듈성, 재사용성, 유지보수성을 크게 향상시킵니다.
