# [Q034] 데이터베이스 트랜잭션 로그와 복구

> 🗄️ Database · 2026-07-02

**Q.** 데이터베이스에서 트랜잭션 로그(Transaction Log)의 역할과 중요성을 설명해주세요. 특히, 트랜잭션 로그가 데이터베이스의 ACID 속성 중 '내구성(Durability)'을 어떻게 보장하는지 설명하고, 시스템 장애 발생 시 데이터베이스가 트랜잭션 로그를 활용하여 데이터를 복구하는 과정(예: Redo, Undo)에 대해 설명해주세요.

## 🧑‍💻 나의 답변

## 데이터베이스 트랜잭션 로그(Transaction Log)의 역할과 중요성

### 1. 핵심 정의

트랜잭션 로그는 데이터베이스에서 발생하는 모든 변경 작업(삽입, 수정, 삭제 등)을 순차적으로 기록하는 파일입니다. 이는 데이터베이스의 현재 상태를 변경하는 모든 작업에 대한 영구적인 기록을 담고 있으며, 주로 WAL(Write-Ahead Logging) 원칙에 따라 동작합니다.

### 2. 동작 원리 및 ACID 속성 중 '내구성(Durability)' 보장

*   **WAL(Write-Ahead Logging) 원칙:** 데이터베이스는 실제 데이터 파일에 변경 사항을 적용하기 전에, 해당 변경 사항을 트랜잭션 로그에 먼저 기록합니다. 이 로그 레코드는 일반적으로 디스크에 동기적으로 기록되어 영구성을 확보합니다.
*   **내구성 보장:** 트랜잭션이 커밋되면, 해당 트랜잭션의 모든 변경 사항이 트랜잭션 로그에 기록되었음이 보장됩니다. 설령 데이터베이스 버퍼 캐시에 있는 변경 사항이 아직 데이터 파일에 기록되지 않은 상태에서 시스템 장애가 발생하더라도, 트랜잭션 로그에 기록된 내용은 손실되지 않습니다. 이 로그를 통해 커밋된 트랜잭션의 변경 사항은 시스템 장애 후에도 복구될 수 있으므로, 내구성이 보장됩니다.

### 3. 시스템 장애 발생 시 데이터 복구 과정 (Redo, Undo)

시스템 장애 발생 시 데이터베이스는 트랜잭션 로그를 활용하여 일관된 상태로 복구됩니다.

*   **Redo (재실행):**
    *   시스템 장애 시점에 커밋되었지만, 아직 데이터 파일에 반영되지 않은 트랜잭션의 변경 사항들을 트랜잭션 로그를 기반으로 데이터 파일에 다시 적용하는 과정입니다.
    *   이를 통해 커밋된 트랜잭션의 내구성을 보장하고, 데이터베이스를 장애 직전의 일관된 상태로 되돌립니다.
*   **Undo (취소):**
    *   시스템 장애 시점에 아직 커밋되지 않았던(진행 중이던) 트랜잭션의 변경 사항들을 트랜잭션 로그를 기반으로 취소하여, 데이터베이스를 해당 트랜잭션이 시작되기 전의 상태로 되돌리는 과정입니다.
    *   이는 ACID 속성 중 '원자성(Atomicity)'을 보장하여, 트랜잭션이 부분적으로만 적용되는 것을 방지합니다.

### 4. 실무에서 주의할 점

*   **성능 영향:** 트랜잭션 로그는 모든 변경 사항을 기록하므로, I/O 성능에 직접적인 영향을 미칩니다. 로그 파일의 위치, 디스크 종류, 쓰기 방식(동기/비동기) 등을 신중하게 고려해야 합니다.
*   **로그 파일 관리:** 트랜잭션 로그 파일은 지속적으로 커지므로, 적절한 크기 관리, 백업, 아카이빙 전략이 필요합니다. 로그가 가득 차면 데이터베이스 작업이 중단될 수 있습니다.
*   **복구 시간:** 로그 파일의 크기가 매우 크거나, 복구해야 할 트랜잭션이 많을 경우 복구 시간이 길어질 수 있습니다. 이는 서비스 가용성에 직접적인 영향을 미치므로, 주기적인 체크포인트(Checkpoint) 설정 등을 통해 복구 시간을 최적화해야 합니다.
