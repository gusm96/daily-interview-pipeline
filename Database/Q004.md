## 데이터베이스 트랜잭션 격리성(Isolation) 및 격리 수준

데이터베이스 트랜잭션의 격리성(Isolation)은 여러 트랜잭션이 동시에 실행될 때, 각 트랜잭션이 마치 독립적으로 실행되는 것처럼 보이게 하는 속성입니다. 즉, 한 트 트랜잭션은 다른 트랜잭션의 중간 결과를 볼 수 없으며, 서로 간섭하지 않고 독립적으로 동작해야 합니다. 이는 동시성 제어(Concurrency Control) 메커니즘을 통해 구현됩니다.

SQL 표준은 다음과 같은 4가지 격리 수준을 정의하며, 각 수준은 동시성 문제 발생 가능성과 성능 간의 트레이드오프를 가집니다.

### 1. Read Uncommitted (가장 낮은 격리 수준)

*   **설명:** 한 트랜잭션이 커밋되지 않은 다른 트랜잭션의 변경 내용을 읽을 수 있습니다. 가장 낮은 격리 수준으로, 동시성 제어 오버헤드가 적어 성능은 좋지만, 동시성 문제가 가장 많이 발생합니다.
*   **발생 가능한 동시성 문제:**
  *   **Dirty Read (더티 리드):** 한 트랜잭션이 아직 커밋되지 않은 다른 트랜잭션의 데이터를 읽는 문제입니다. 만약 변경 사항이 롤백되면, 첫 번째 트랜잭션은 존재하지 않는 데이터를 읽은 셈이 됩니다.
      *   **예시:**
          1.  **T1:** 계좌 잔액 1000원에서 500원으로 변경 (아직 커밋 안 됨)
          2.  **T2:** 계좌 잔액 조회 -> 500원 (T1의 커밋되지 않은 변경 사항 읽음)
          3.  **T1:** 어떤 이유로 롤백되어 잔액이 다시 1000원으로 돌아감
          4.  **T2:** 500원이라는 잘못된 정보를 기반으로 다음 작업을 진행

### 2. Read Committed

*   **설명:** 한 트랜잭션은 커밋된 데이터만 읽을 수 있습니다. Dirty Read는 방지하지만, 트랜잭션 내에서 동일한 데이터를 여러 번 읽을 때 다른 결과가 나올 수 있습니다. 대부분의 데이터베이스 시스템의 기본 격리 수준입니다.
*   **발생 가능한 동시성 문제:**
  *   **Non-repeatable Read (반복 불가능한 읽기):** 한 트랜잭션 내에서 동일한 쿼리를 두 번 실행했을 때, 그 사이에 다른 트랜잭션이 데이터를 수정하고 커밋하여 두 쿼리의 결과가 달라지는 문제입니다.
      *   **예시:**
          1.  **T1:** 사용자 '홍길동'의 정보를 조회 -> 이름: 홍길동, 나이: 30
          2.  **T2:** 사용자 '홍길동'의 나이를 30에서 31로 변경하고 커밋
          3.  **T1:** 다시 사용자 '홍길동'의 정보를 조회 -> 이름: 홍길동, 나이: 31 (이전과 다른 결과)

### 3. Repeatable Read

*   **설명:** 한 트랜잭션 내에서 동일한 쿼리를 여러 번 실행했을 때 항상 동일한 결과를 보장합니다. Non-repeatable Read를 방지합니다. 일반적으로 읽기 잠금(Shared Lock)을 사용하여 구현됩니다.
*   **발생 가능한 동시성 문제:**
  *   **Phantom Read (유령 읽기):** 한 트랜잭션 내에서 특정 조건에 해당하는 레코드 집합을 조회했을 때, 그 사이에 다른 트랜잭션이 해당 조건에 맞는 새로운 레코드를 삽입하고 커밋하여, 다시 조회했을 때 이전에 없던 레코드가 나타나는 문제입니다.
      *   **예시:**
          1.  **T1:** '부서 A'에 속한 직원 수를 조회 -> 5명
          2.  **T2:** '부서 A'에 새로운 직원 1명을 추가하고 커밋
          3.  **T1:** 다시 '부서 A'에 속한 직원 수를 조회 -> 6명 (이전에 없던 '유령' 레코드가 나타남)

### 4. Serializable (가장 높은 격리 수준)

*   **설명:** 가장 엄격한 격리 수준으로, 모든 동시성 문제를 방지합니다. 트랜잭션들이 순차적으로 실행되는 것과 동일한 결과를 보장합니다. 일반적으로 범위 잠금(Range Lock)이나 다중 버전 동시성 제어(MVCC)를 통해 구현됩니다.
*   **발생 가능한 동시성 문제:** 모든 동시성 문제를 방지합니다.
*   **단점:** 동시성 제어 오버헤드가 가장 커서 성능 저하가 발생할 수 있습니다. 동시성 처리량이 중요한 시스템에서는 신중하게 고려해야 합니다.

### 요약 및 선택 가이드

| 격리 수준         | Dirty Read 방지 | Non-repeatable Read 방지 | Phantom Read 방지 | 성능 (상대적) |
| :---------------- | :-------------- | :----------------------- | :---------------- | :------------ |
| Read Uncommitted  | X               | X                        | X                 | 높음          |
| Read Committed    | O               | X                        | X                 | 중간          |
| Repeatable Read   | O               | O                        | X                 | 중간-낮음     |
| Serializable      | O               | O                        | O                 | 낮음          |

대부분의 애플리케이션에서는 **Read Committed** 격리 수준이 기본으로 사용되며, 이는 Dirty Read를 방지하면서 적절한 성능을 제공합니다. 특정 비즈니스 로직에서 Non-repeatable Read나 Phantom Read가 심각한 문제를 야기할 경우, **Repeatable Read** 또는 **Serializable**을 고려할 수 있지만, 이 경우 성능 저하를 감수해야 합니다. 따라서 시스템의 요구사항과 동시성 문제의 심각도를 고려하여 적절한 격리 수준을 선택하는 것이 중요합니다.
