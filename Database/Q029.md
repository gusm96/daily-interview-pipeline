## 데이터베이스 락(Lock)의 종류와 특징, 데드락(Deadlock) 관리 전략

데이터베이스에서 동시성 제어를 위해 사용되는 락은 여러 트랜잭션이 동시에 데이터에 접근할 때 데이터의 일관성과 무결성을 유지하기 위한 메커니즘입니다.

### 락의 종류와 특징

1.  **공유 락 (Shared Lock / S-Lock)**
    *   **특징:** 데이터를 읽을 때 사용됩니다. 여러 트랜잭션이 동시에 동일한 데이터에 공유 락을 걸 수 있습니다. 즉, 여러 사용자가 동시에 같은 데이터를 읽는 것은 허용하지만, 해당 데이터에 쓰기 락(배타 락)이 걸리는 것은 방지합니다.
    *   **예시:** `SELECT` 문 실행 시 자동으로 걸립니다.

2.  **배타 락 (Exclusive Lock / X-Lock)**
    *   **특징:** 데이터를 변경(쓰기)할 때 사용됩니다. 특정 데이터에 배타 락이 걸리면, 다른 어떤 트랜잭션도 해당 데이터에 공유 락이나 배타 락을 걸 수 없습니다. 즉, 한 번에 하나의 트랜잭션만 데이터를 수정할 수 있도록 보장합니다.
    *   **예시:** `INSERT`, `UPDATE`, `DELETE` 문 실행 시 자동으로 걸립니다.

3.  **행 락 (Row-Level Lock)**
    *   **특징:** 데이터베이스 테이블의 특정 행(레코드)에만 락을 겁니다. 가장 세분화된 락으로, 동시성을 최대로 높일 수 있습니다. 한 트랜잭션이 특정 행을 수정하더라도, 다른 트랜잭션은 다른 행을 수정하거나 읽을 수 있습니다.
    *   **장점:** 동시성(Concurrency)이 높습니다.
    *   **단점:** 락 관리 오버헤드가 상대적으로 큽니다.

4.  **테이블 락 (Table-Level Lock)**
    *   **특징:** 데이터베이스 테이블 전체에 락을 겁니다. 테이블에 테이블 락이 걸리면, 해당 테이블의 어떤 행에도 다른 트랜잭션이 접근할 수 없습니다.
    *   **장점:** 락 관리 오버헤드가 적습니다.
    *   **단점:** 동시성(Concurrency)이 매우 낮아집니다. 주로 테이블 구조 변경(DDL) 시 사용되거나, 특정 상황에서 성능 최적화를 위해 명시적으로 사용될 수 있습니다.

### 데드락(Deadlock)

데드락은 두 개 이상의 트랜잭션이 서로가 가지고 있는 락을 기다리면서 무한정 대기하는 상태를 말합니다.

#### 데드락이 발생하는 조건 (네 가지 필수 조건)

1.  **상호 배제 (Mutual Exclusion):** 최소한 하나의 자원(데이터)은 한 번에 하나의 트랜잭션만 사용할 수 있어야 합니다. (락의 본질)
2.  **점유와 대기 (Hold and Wait):** 자원을 점유하고 있는 트랜잭션이 다른 자원을 요청하고 대기해야 합니다.
3.  **비선점 (No Preemption):** 이미 할당된 자원은 강제로 빼앗을 수 없으며, 자원을 점유한 트랜잭션이 스스로 반납할 때까지 기다려야 합니다.
4.  **순환 대기 (Circular Wait):** 트랜잭션들이 자원 대기 그래프에서 순환 형태를 이루어야 합니다. (예: T1이 A를 점유하고 B를 대기, T2가 B를 점유하고 A를 대기)

#### 데드락 방지 또는 해결 전략

1.  **타임아웃 (Timeout):**
    *   **설명:** 락을 획득하기 위해 일정 시간 이상 대기하면, 대기를 중단하고 에러를 발생시켜 트랜잭션을 롤백하는 방식입니다.
    *   **장점:** 구현이 비교적 간단하고, 데드락 상태를 강제로 해제할 수 있습니다.
    *   **단점:** 데드락이 아닌 단순한 부하로 인한 지연일 경우에도 트랜잭션이 롤백될 수 있습니다.

2.  **락 순서 지정 (Lock Ordering / Resource Ordering):**
    *   **설명:** 모든 트랜잭션이 락을 획득할 때 항상 동일한 순서로 자원(테이블, 행 등)에 락을 걸도록 규칙을 정하는 방식입니다.
    *   **장점:** 순환 대기 조건을 원천적으로 방지하여 데드락 발생 가능성을 크게 줄입니다.
    *   **단점:** 시스템 설계 단계에서 모든 자원에 대한 락 순서를 정의하고, 이를 모든 트랜잭션이 준수하도록 구현하는 것이 복잡하고 어렵습니다. 특히 동적으로 자원에 접근하는 경우 적용하기 어렵습니다.

3.  **데드락 감지 및 복구 (Deadlock Detection and Recovery):**
    *   **설명:**
        *   **감지:** 데이터베이스 시스템이 주기적으로 자원 할당 그래프를 검사하여 순환 대기 상태(데드락)를 감지합니다.
        *   **복구:** 데드락이 감지되면, 데드락에 연루된 트랜잭션 중 하나(희생자, Victim)를 선택하여 강제로 롤백(Abort)하고, 해당 트랜잭션이 점유하고 있던 락을 해제하여 다른 트랜잭션들이 진행할 수 있도록 합니다. 희생자 선택 기준은 보통 가장 적은 작업을 수행했거나, 가장 적은 락을 보유한 트랜잭션 등이 될 수 있습니다.
    *   **장점:** 데드락 발생을 허용하되, 발생 시 효과적으로 해결할 수 있습니다.
    *   **단점:** 데드락 감지 및 복구 과정 자체에 오버헤드가 발생하며, 롤백된 트랜잭션은 재시작해야 합니다. 대부분의 관계형 데이터베이스 관리 시스템(RDBMS)이 이 방식을 사용합니다.

이러한 락의 종류와 데드락 관리 전략을 이해하는 것은 데이터베이스 시스템의 성능과 안정성을 최적화하는 데 매우 중요합니다.
