## 파티셔닝(Partitioning) 전략 및 고려사항

### 1. 핵심 정의

파티셔닝은 대용량 데이터베이스에서 테이블이나 인덱스를 더 작고 관리하기 쉬운 여러 개의 논리적인 조각(파티션)으로 나누는 기법입니다. 이를 통해 데이터베이스의 성능, 가용성, 관리 용이성을 향상시킬 수 있습니다.

### 2. 동작 원리 및 전략별 장단점

파티셔닝 전략은 데이터를 어떤 기준으로 나눌지에 따라 달라지며, 주요 전략은 다음과 같습니다.

*   **Range Partitioning (범위 파티셔닝)**
    *   **동작 원리:** 특정 컬럼(예: 날짜, 숫자 범위)의 값 범위를 기준으로 데이터를 나눕니다.
    *   **장점:**
        *   특정 범위의 데이터를 조회하거나 삭제할 때 성능 향상에 유리합니다 (파티션 프루닝).
        *   데이터 보관 정책(예: 오래된 데이터 아카이빙) 구현에 용이합니다.
    *   **단점:**
        *   데이터가 특정 범위에 집중될 경우 핫스팟(Hotspot)이 발생할 수 있습니다.
        *   파티션 키 선택이 중요하며, 잘못 선택 시 데이터 불균형을 초래할 수 있습니다.

*   **List Partitioning (리스트 파티셔닝)**
    *   **동작 원리:** 특정 컬럼의 이산적인 값 목록을 기준으로 데이터를 나눕니다.
    *   **장점:**
        *   특정 값 그룹에 대한 쿼리 성능 향상에 유리합니다.
        *   데이터의 논리적인 그룹화가 명확할 때 유용합니다 (예: 지역, 상태 코드).
    *   **단점:**
        *   파티션 키의 값이 미리 정의되어야 하며, 새로운 값이 추가될 경우 파티션 정의를 수정해야 할 수 있습니다.
        *   정의되지 않은 값에 대한 처리가 필요합니다 (DEFAULT 파티션).

*   **Hash Partitioning (해시 파티셔닝)**
    *   **동작 원리:** 파티션 키의 해시 값을 계산하여 미리 정의된 파티션 수만큼 균등하게 데이터를 분산시킵니다.
    *   **장점:**
        *   데이터를 파티션 간에 비교적 균등하게 분산시켜 핫스팟 발생 가능성을 줄입니다.
        *   특정 범위나 목록에 대한 쿼리보다는 전체 데이터에 대한 병렬 처리 성능 향상에 유리합니다.
    *   **단점:**
        *   특정 파티션에 대한 직접적인 접근이 어렵고, 파티션 프루닝 효과가 제한적입니다.
        *   파티션 수를 변경하기 어렵고, 변경 시 데이터 재분배가 필요할 수 있습니다.

### 3. 실무 적용 시 주의할 점

*   **파티션 키(Partition Key) 선정:**
    *   쿼리 패턴, 데이터 분포, 데이터 증가 추이를 고려하여 신중하게 선정해야 합니다.
    *   파티션 키는 자주 조회되거나 범위 검색에 사용되는 컬럼이 적합합니다.
    *   파티션 키가 너무 적은 카디널리티를 가지면 데이터 불균형을 초래할 수 있습니다.
*   **파티션 수 및 크기:**
    *   너무 많은 파티션은 관리 오버헤드를 증가시키고, 너무 적은 파티션은 파티셔닝의 이점을 상쇄시킬 수 있습니다.
    *   일반적으로 파티션당 데이터 크기가 너무 커지지 않도록 적절한 수를 유지하는 것이 중요합니다.
*   **유지보수 및 관리:**
    *   새로운 파티션 추가, 오래된 파티션 삭제/아카이빙 등 파티션 라이프사이클 관리가 필요합니다.
    *   파티션 재구성(Reorganization)은 많은 리소스를 소모하므로 신중하게 계획해야 합니다.
*   **글로벌 인덱스 vs 로컬 인덱스:**
    *   파티셔닝된 테이블에 인덱스를 생성할 때, 모든 파티션에 걸쳐 있는 글로벌 인덱스와 각 파티션에 독립적으로 생성되는 로컬 인덱스 중 적절한 것을 선택해야 합니다.
    *   로컬 인덱스는 파티션 독립적인 관리가 가능하지만, 글로벌 인덱스는 파티션 변경 시 재구축 비용이 발생할 수 있습니다.
*   **애플리케이션 변경:**
    *   파티셔닝 적용 시 애플리케이션의 쿼리 로직이 파티션 프루닝을 활용할 수 있도록 최적화되어야 합니다.
    *   파티션 키를 쿼리에 포함시켜 불필요한 파티션 스캔을 방지하는 것이 중요합니다.
