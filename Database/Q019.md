# [Q019] NoSQL 데이터베이스 유형과 선택

> 🗄️ Database · 2026-06-27

**Q.** 관계형 데이터베이스(RDBMS) 외에 NoSQL 데이터베이스가 등장하게 된 배경과 목적을 설명해주세요. 주요 NoSQL 데이터베이스 유형(예: Key-Value, Document, Column-Family, Graph)을 각각 설명하고, 각 유형이 어떤 데이터 모델과 사용 사례에 적합한지 구체적인 예시를 들어 설명해주세요. 또한, 특정 비즈니스 요구사항에 맞춰 NoSQL 데이터베이스를 선택할 때 고려해야 할 주요 요소들에 대해 설명해주세요.

## 🧑‍💻 나의 답변

## NoSQL 데이터베이스 등장 배경, 유형, 선택 고려사항

### 1. NoSQL 데이터베이스 등장 배경 및 목적

관계형 데이터베이스(RDBMS)는 데이터의 일관성과 무결성을 보장하는 데 탁월하지만, 웹 서비스의 폭발적인 성장과 함께 다음과 같은 한계에 직면했습니다.

*   **확장성(Scalability) 한계:** RDBMS는 수직적 확장(서버 성능 향상)에는 용이하지만, 수평적 확장(서버 증설)에는 복잡성과 비용이 많이 듭니다. 대규모 트래픽과 데이터 처리에 어려움이 있었습니다.
*   **유연성(Flexibility) 부족:** RDBMS는 엄격한 스키마를 요구하여 데이터 모델 변경이 어렵고, 비정형/반정형 데이터 저장에 비효율적이었습니다.
*   **성능(Performance) 저하:** 대량의 데이터 조인 연산은 성능 저하를 야기하며, 특정 유형의 데이터(예: 소셜 그래프 데이터) 처리에는 비효율적이었습니다.

이러한 RDBMS의 한계를 극복하고, **대규모 분산 환경에서의 확장성, 유연성, 성능**을 최우선으로 하여 등장한 것이 NoSQL(Not only SQL) 데이터베이스입니다. NoSQL의 주요 목적은 다음과 같습니다.

*   **수평적 확장성:** 저렴한 서버를 추가하여 손쉽게 시스템 용량을 확장합니다.
*   **유연한 스키마:** 스키마 없이 데이터를 저장하거나, 동적인 스키마를 지원하여 데이터 모델 변경에 유연하게 대응합니다.
*   **고성능:** 특정 데이터 모델에 최적화된 저장 방식을 통해 빠른 읽기/쓰기 성능을 제공합니다.
*   **다양한 데이터 유형 지원:** 정형, 반정형, 비정형 데이터를 효율적으로 저장하고 처리합니다.

### 2. 주요 NoSQL 데이터베이스 유형 및 사용 사례

| 유형 | 설명 | 데이터 모델 | 사용 사례 예시 |
| :--- | :--- | :--- | :--- |
| **Key-Value Store** | 가장 단순한 형태. 고유한 키(Key)와 해당 키에 매핑되는 값(Value)으로 데이터를 저장. | 단순한 해시 테이블 형태 | - **세션 관리:** 사용자 세션 정보 (예: Redis) <br> - **캐싱:** 자주 접근하는 데이터 캐싱 (예: Redis, Memcached) <br> - **장바구니:** 사용자별 장바구니 내용 |
| **Document Store** | 데이터를 문서(Document) 형태로 저장. 문서는 JSON, BSON, XML 등 다양한 형식으로 표현될 수 있으며, 스키마가 유연함. | 중첩된 구조를 갖는 문서 (JSON과 유사) | - **콘텐츠 관리 시스템(CMS):** 블로그 게시물, 기사, 제품 정보 (예: MongoDB) <br> - **카탈로그:** 전자상거래 제품 카탈로그 <br> - **사용자 프로필:** 복잡하고 유동적인 사용자 프로필 정보 |
| **Column-Family Store** | 데이터를 행(Row)과 열(Column)의 집합으로 구성하지만, RDBMS와 달리 열 그룹(Column Family)을 유연하게 정의하고, 각 행마다 다른 열을 가질 수 있음. | 희소 행렬(Sparse Matrix) 형태, 열 그룹 | - **빅데이터 분석:** 대규모 시계열 데이터, 로그 데이터 (예: Apache Cassandra, HBase) <br> - **실시간 추천 시스템:** 사용자 행동 기반 추천 데이터 <br> - **IoT 데이터:** 센서 데이터 저장 및 분석 |
| **Graph Database** | 데이터 간의 관계(Relationship)를 노드(Node)와 엣지(Edge)로 표현하여 저장. 관계 질의에 최적화되어 있음. | 노드(개체), 엣지(관계), 속성 | - **소셜 네트워크:** 친구 관계, 팔로우 관계 (예: Neo4j) <br> - **추천 시스템:** 사용자-아이템 관계 기반 추천 <br> - **사기 탐지:** 복잡한 거래 관계 분석 <br> - **지식 그래프:** 엔티티 간의 복잡한 관계 표현 |

### 3. NoSQL 데이터베이스 선택 시 고려해야 할 주요 요소

특정 비즈니스 요구사항에 맞춰 NoSQL 데이터베이스를 선택할 때는 다음과 같은 요소들을 종합적으로 고려해야 합니다.

1.  **데이터 모델 및 접근 패턴:**
    *   **데이터 구조:** 데이터가 정형적인가, 반정형적인가, 비정형적인가? 계층적인가, 관계 중심적인가?
    *   **주요 질의 유형:** 주로 키-값으로 조회하는가, 복잡한 문서 구조를 조회하는가, 관계를 탐색하는가, 특정 열 그룹을 조회하는가?
    *   **데이터 변경 빈도:** 데이터가 자주 변경되는가, 한 번 저장되면 거의 변경되지 않는가?

2.  **확장성 요구사항:**
    *   **예상 데이터 볼륨:** 현재 및 미래에 예상되는 데이터의 양은 어느 정도인가?
    *   **예상 트래픽:** 초당 읽기/쓰기 요청 수는 어느 정도인가?
    *   **수평적 확장 필요성:** 시스템이 성장함에 따라 서버를 쉽게 추가하여 용량을 늘려야 하는가?

3.  **일관성 및 가용성 요구사항 (CAP 이론):**
    *   **CAP 이론 이해:** NoSQL은 분산 시스템에서 일관성(Consistency), 가용성(Availability), 분할 내성(Partition Tolerance) 중 두 가지만 만족할 수 있다는 CAP 이론을 따릅니다.
    *   **비즈니스 중요도:** 데이터의 최신성이 매우 중요한가(일관성)? 아니면 항상 서비스가 가능해야 하는가(가용성)? (예: 금융 거래는 일관성, 소셜 미디어 피드는 가용성)

4.  **성능 요구사항:**
    *   **응답 시간:** 특정 작업에 대한 허용 가능한 응답 시간은 어느 정도인가?
    *   **처리량:** 단위 시간당 처리해야 할 요청 수는 어느 정도인가?

5.  **운영 및 관리 용이성:**
    *   **개발자 숙련도:** 팀 내에서 특정 NoSQL DB에 대한 경험이 있는가?
    *   **커뮤니티 및 지원:** 활발한 커뮤니티와 충분한 기술 지원이 제공되는가?
    *   **클라우드 서비스 지원:** 클라우드 환경에서 관리형 서비스로 제공되는가? (예: AWS DynamoDB, Azure Cosmos DB)
    *   **모니터링 및 백업/복구:** 운영에 필요한 도구와 기능이 잘 갖춰져 있는가?

6.  **비용:**
    *   **하드웨어 비용:** 필요한 서버 사양 및 대수는?
    *   **라이선스 비용:** 오픈소스인가, 상용 라이선스인가?
    *   **운영 인력 비용:** 관리 및 유지보수에 필요한 인력은?

이러한 요소들을 종합적으로 고려하여 비즈니스 요구사항에 가장 적합한 NoSQL 데이터베이스 유형과 특정 제품을 선택해야 합니다. 예를 들어, 대규모 사용자 세션 관리가 필요하다면 Key-Value Store인 Redis를, 유연한 스키마로 제품 카탈로그를 관리해야 한다면 Document Store인 MongoDB를, 대규모 시계열 데이터를 저장하고 분석해야 한다면 Column-Family Store인 Cassandra를 고려할 수 있습니다.
