# [Q009] 인덱스 동작 원리와 최적화

> 🗄️ Database · 2026-06-19

**Q.** 데이터베이스 인덱스는 쿼리 성능 최적화에 필수적인 요소입니다. B-Tree 인덱스의 기본적인 동작 원리와 구조를 설명하고, 인덱스를 효과적으로 활용하기 위한 쿼리 작성 및 인덱스 설계 전략에 대해 설명해주세요. 또한, 인덱스 사용 시 주의해야 할 점(예: 인덱스 스캔 vs 테이블 스캔, 인덱스 추가 비용)은 무엇인지 함께 설명해주세요.

## 🧑‍💻 나의 답변

## 데이터베이스 B-Tree 인덱스 동작 원리, 활용 전략 및 주의사항

### 1. B-Tree 인덱스의 기본적인 동작 원리와 구조

B-Tree 인덱스는 데이터베이스에서 가장 널리 사용되는 인덱스 구조입니다. "Balanced Tree"의 약자로, 모든 리프 노드(데이터를 가리키는 노드)가 동일한 깊이를 가지도록 균형을 유지하는 트리 구조입니다.

**동작 원리:**

1. **루트 노드부터 탐색:** 쿼리 시, B-Tree의 루트 노드부터 시작하여 검색하려는 값과 노드의 키 값을 비교합니다.
2. **자식 노드 이동:** 비교 결과에 따라 적절한 자식 노드로 이동합니다. (예: 검색 값이 현재 노드의 키 값보다 작으면 왼쪽 자식, 크면 오른쪽 자식)
3. **리프 노드 도달:** 이 과정을 반복하여 최종적으로 검색하려는 값이 저장된 리프 노드에 도달합니다.
4. **데이터 레코드 포인터 획득:** 리프 노드에는 실제 데이터 레코드의 물리적 주소(RID 또는 Primary Key)가 저장되어 있어, 이를 통해 빠르게 해당 데이터를 찾아갈 수 있습니다.

**구조:**

*   **노드(Node):** B-Tree를 구성하는 기본 단위로, 여러 개의 키(Key)와 자식 노드에 대한 포인터(Pointer)를 가집니다.
*   **키(Key):** 인덱스를 구성하는 컬럼의 값입니다.
*   **포인터(Pointer):** 다음 레벨의 자식 노드 또는 실제 데이터 레코드를 가리킵니다.
*   **루트 노드(Root Node):** 트리의 최상단 노드입니다.
*   **내부 노드(Internal Node):** 루트 노드와 리프 노드 사이에 있는 노드들입니다.
*   **리프 노드(Leaf Node):** 트리의 최하단 노드로, 실제 데이터 레코드의 위치 정보(RID 또는 Primary Key)를 저장합니다. 리프 노드들은 보통 서로 연결되어 있어 범위 검색에 효율적입니다.

### 2. 인덱스를 효과적으로 활용하기 위한 쿼리 작성 및 인덱스 설계 전략

**쿼리 작성 전략:**

*   **WHERE 절 활용:** 인덱스는 `WHERE` 절의 조건(등호, 부등호, `LIKE` '값%', `BETWEEN`, `IN`)에 사용될 때 가장 효과적입니다.
*   **선택도 높은 컬럼 우선:** `WHERE` 절에 여러 조건이 있다면, 데이터의 분포가 넓어 필터링 효과가 큰(선택도가 높은) 컬럼을 먼저 인덱스에 포함하는 것이 좋습니다.
*   **`ORDER BY`, `GROUP BY` 활용:** `ORDER BY`나 `GROUP BY` 절에 사용되는 컬럼에 인덱스가 있으면 정렬 및 그룹화 작업의 성능을 향상시킬 수 있습니다.
*   **커버링 인덱스 활용:** 쿼리에서 필요한 모든 컬럼이 인덱스 자체에 포함되어 있다면, 데이터 테이블에 접근할 필요 없이 인덱스만으로 쿼리를 처리할 수 있어 성능이 크게 향상됩니다. (인덱스 온리 스캔)
*   **함수 사용 지양:** `WHERE` 절에 인덱스 컬럼에 함수를 적용하면 인덱스를 사용하지 못하고 테이블 스캔이 발생할 수 있습니다. (예: `WHERE YEAR(date_column) = 2023` 대신 `WHERE date_column BETWEEN '2023-01-01' AND '2023-12-31'`)
*   **`LIKE` '%값' 지양:** `LIKE` 연산자에서 와일드카드(%)가 앞에 오면 인덱스를 사용할 수 없습니다.

**인덱스 설계 전략:**

*   **자주 검색되는 컬럼:** `WHERE`, `JOIN`, `ORDER BY`, `GROUP BY` 절에 자주 사용되는 컬럼에 인덱스를 생성합니다.
*   **카디널리티 고려:** 컬럼의 카디널리티(고유한 값의 수)가 높을수록 인덱스의 효율이 좋습니다. (예: 성별과 같이 카디널리티가 낮은 컬럼은 인덱스 효율이 떨어질 수 있습니다.)
*   **복합 인덱스 활용:** 여러 컬럼을 조합하여 인덱스를 생성할 때, 쿼리의 `WHERE` 절 조건 순서와 일치하도록 컬럼 순서를 지정하는 것이 중요합니다. (좌측 접두사 규칙)
*   **불필요한 인덱스 제거:** 인덱스는 추가적인 저장 공간과 관리 비용을 발생시키므로, 사용되지 않거나 효율이 낮은 인덱스는 제거하여 오버헤드를 줄입니다.
*   **쓰기 작업 빈도 고려:** `INSERT`, `UPDATE`, `DELETE` 작업이 빈번한 테이블에는 인덱스 추가에 신중해야 합니다. (아래 주의사항 참고)
*   **클러스터형 인덱스 활용 (Primary Key):** 대부분의 DBMS에서 Primary Key는 자동으로 클러스터형 인덱스로 생성됩니다. 클러스터형 인덱스는 데이터 레코드 자체를 인덱스 순서에 맞춰 물리적으로 저장하므로, 범위 검색 및 정렬에 매우 효율적입니다.

### 3. 인덱스 사용 시 주의해야 할 점

*   **인덱스 스캔 vs 테이블 스캔:**
    *   **인덱스 스캔:** 인덱스를 사용하여 필요한 데이터를 찾는 방식입니다. 일반적으로 테이블 스캔보다 훨씬 빠릅니다. 특히, `WHERE` 절 조건에 의해 필터링되는 데이터의 양이 적을 때 효율적입니다.
    *   **테이블 스캔 (Full Table Scan):** 인덱스를 사용하지 않고 테이블의 모든 레코드를 처음부터 끝까지 읽는 방식입니다. `WHERE` 절 조건이 없거나, 인덱스를 사용할 수 없는 경우, 또는 쿼리 결과로 반환되는 데이터의 양이 너무 많아 인덱스를 사용하는 것보다 테이블 전체를 읽는 것이 더 효율적이라고 옵티마이저가 판단할 때 발생합니다. 테이블 스캔은 대량의 데이터를 처리할 때 성능 저하의 주범이 될 수 있습니다.

*   **인덱스 추가 비용 (오버헤드):**
    *   **저장 공간:** 인덱스 자체도 데이터를 저장하므로 추가적인 디스크 공간을 차지합니다.
    *   **쓰기 성능 저하:** `INSERT`, `UPDATE`, `DELETE`와 같은 데이터 변경 작업이 발생할 때마다, 데이터 테이블뿐만 아니라 해당 테이블의 모든 인덱스도 함께 갱신되어야 합니다. 이 과정에서 추가적인 I/O 및 CPU 자원이 소모되므로, 쓰기 작업이 빈번한 테이블에 너무 많은 인덱스를 추가하면 오히려 전체적인 성능이 저하될 수 있습니다.
    *   **관리 비용:** 인덱스가 많아질수록 데이터베이스 관리 시스템(DBMS)이 쿼리 실행 계획을 수립할 때 고려해야 할 경우의 수가 늘어나 최적의 인덱스를 선택하는 데 시간이 더 걸릴 수 있습니다.

**결론적으로, 인덱스는 쿼리 성능 최적화에 강력한 도구이지만, 무분별한 사용은 오히려 독이 될 수 있습니다. 쿼리 패턴과 데이터 특성을 면밀히 분석하여 필요한 곳에만 적절하게 인덱스를 설계하고 관리하는 것이 중요합니다.**
