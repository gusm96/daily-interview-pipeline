## 마이크로서비스 아키텍처(MSA) 서비스 디스커버리 모범 답안

서비스 디스커버리는 MSA에서 동적으로 변화하는 서비스 인스턴스를 찾고 통신하는 핵심 메커니즘입니다. 크게 클라이언트 측과 서버 측 서비스 디스커버리로 나눌 수 있습니다.

### 1. 클라이언트 측 서비스 디스커버리 (Client-side Service Discovery)

**개념:** 클라이언트(서비스를 호출하는 측)가 직접 서비스 레지스트리에서 호출할 서비스 인스턴스의 네트워크 위치를 조회하고, 해당 정보를 바탕으로 직접 서비스 인스턴스에 요청을 보냅니다.

**장점:**
*   **단순성:** 서버 측에 추가적인 프록시 계층이 필요 없어 아키텍처가 비교적 단순합니다.
*   **유연성:** 클라이언트가 직접 로드 밸런싱 알고리즘(라운드 로빈, 최소 연결 등)을 선택하거나 구현할 수 있습니다.
*   **성능:** 중간 계층이 없어 Latency가 적을 수 있습니다.

**단점:**
*   **클라이언트 복잡성:** 모든 클라이언트가 서비스 디스커버리 로직(레지스트리 조회, 로드 밸런싱)을 내장해야 하므로 클라이언트 개발 및 유지보수가 복잡해집니다.
*   **언어 종속성:** 각 클라이언트 언어/프레임워크별로 디스커버리 라이브러리를 구현해야 합니다.
*   **업데이트 어려움:** 디스커버리 로직 변경 시 모든 클라이언트를 업데이트해야 합니다.

**구현 방식:**
*   **Eureka (Netflix OSS):** Spring Cloud 환경에서 널리 사용되며, 서비스 인스턴스가 Eureka 서버에 스스로 등록하고 클라이언트는 Eureka 서버에서 인스턴스 정보를 가져옵니다.
*   **Consul (HashiCorp):** 서비스 등록/조회 외에 헬스 체크, 키-값 저장소, 분산 락 등 다양한 기능을 제공합니다. 서비스 인스턴스가 Consul 에이전트를 통해 등록하고 클라이언트는 Consul API를 통해 조회합니다.

### 2. 서버 측 서비스 디스커버리 (Server-side Service Discovery)

**개념:** 클라이언트가 서비스 인스턴스의 네트워크 위치를 직접 알 필요 없이, 중간 프록시(로드 밸런서)를 통해 서비스에 요청을 보냅니다. 이 프록시가 서비스 레지스트리에서 인스턴스 정보를 조회하여 요청을 적절한 인스턴스로 라우팅합니다.

**장점:**
*   **클라이언트 단순성:** 클라이언트는 프록시의 고정된 주소만 알면 되므로 디스커버리 로직이 필요 없어 개발이 단순합니다.
*   **언어 독립성:** 클라이언트 언어에 관계없이 동일한 방식으로 서비스 호출이 가능합니다.
*   **중앙 집중 관리:** 로드 밸런싱, 헬스 체크, 라우팅 정책 등을 프록시 계층에서 중앙 집중적으로 관리할 수 있습니다.

**단점:**
*   **추가 인프라:** 프록시(로드 밸런서) 계층이 추가되어 아키텍처가 복잡해지고 관리 포인트가 늘어납니다.
*   **성능 오버헤드:** 프록시를 거치므로 Latency가 증가할 수 있습니다.
*   **단일 장애점 가능성:** 프록시 계층이 단일 장애점이 될 수 있으므로 고가용성 구성이 필수적입니다.

**구현 방식:**
*   **Kubernetes Service:** Kubernetes는 `Service`라는 추상화 계층을 제공하여 파드(Pod) 그룹에 대한 안정적인 네트워크 엔드포인트를 제공합니다. 클라이언트는 `Service`의 DNS 이름으로 접근하고, Kubernetes는 내부적으로 `kube-proxy`를 통해 요청을 적절한 파드로 라우팅합니다.
*   **AWS ELB (Elastic Load Balancer):** EC2 인스턴스나 컨테이너 서비스 앞에 ELB를 두어 트래픽을 분산합니다. ELB는 등록된 인스턴스의 헬스 체크를 수행하고 정상 인스턴스로만 트래픽을 보냅니다.

### 3. AWS 환경에서의 서비스 디스커버리 구현

AWS는 MSA 환경에서 서비스 디스커버리를 효과적으로 구현할 수 있는 다양한 서비스를 제공합니다.

*   **AWS Cloud Map:**
    *   **개념:** 모든 AWS 리소스(EC2, ECS, EKS, Lambda 등)에 대한 사용자 정의 이름 공간을 생성하고, 해당 리소스의 위치를 등록하여 DNS 기반 또는 API 기반으로 검색할 수 있게 해주는 서비스입니다.
    *   **활용 방안:**
        *   **ECS/EKS 서비스 디스커버리:** ECS/EKS 서비스가 시작될 때 자동으로 Cloud Map에 등록되도록 설정할 수 있습니다. 다른 서비스는 Cloud Map에 등록된 DNS 이름을 통해 해당 서비스를 호출하거나, Cloud Map API를 통해 서비스 인스턴스 목록을 조회하여 직접 연결할 수 있습니다.
        *   **내부 마이크로서비스 간 통신:** 예를 들어, `order-service`가 `product-service`를 호출해야 할 때, `product-service.internal`과 같은 Cloud Map에 등록된 DNS 이름을 사용하여 쉽게 통신할 수 있습니다. Cloud Map은 헬스 체크를 통해 비정상 인스턴스를 자동으로 제거하여 안정적인 통신을 보장합니다.
        *   **Lambda 함수 호출:** Lambda 함수가 다른 마이크로서비스를 호출할 때 Cloud Map을 통해 동적으로 서비스 엔드포인트를 찾아 호출할 수 있습니다.

*   **ECS Service Discovery (AWS Cloud Map 기반):**
    *   **개념:** AWS ECS(Elastic Container Service)에서 태스크(Task) 정의 시 서비스 디스커버리 설정을 포함하여, 해당 서비스의 태스크 인스턴스가 자동으로 AWS Cloud Map에 등록되도록 하는 기능입니다. 이는 사실상 Cloud Map을 ECS에 통합하여 사용하는 방식입니다.
    *   **활용 방안:**
        *   **ECS 서비스 간 통신:** ECS 클러스터 내에서 `frontend-service`가 `backend-api-service`를 호출해야 할 때, `backend-api-service`의 ECS 서비스 정의에 Cloud Map을 연동하면, `frontend-service`는 `backend-api-service.my-namespace`와 같은 DNS 이름을 통해 `backend-api-service`의 태스크 인스턴스에 접근할 수 있습니다. ECS는 태스크의 라이프사이클에 맞춰 Cloud Map 레코드를 자동으로 업데이트합니다.
        *   **로드 밸런싱 및 헬스 체크:** ECS 서비스 디스커버리는 Cloud Map의 헬스 체크 기능을 활용하여 비정상 태스크를 자동으로 DNS 레코드에서 제거하고, DNS 라운드 로빈을 통해 정상 태스크로 트래픽을 분산합니다. 이를 통해 별도의 로드 밸런서 없이도 내부 서비스 간의 기본적인 로드 밸런싱을 구현할 수 있습니다.

**결론적으로,** 클라이언트 측 디스커버리는 클라이언트의 유연성과 성능에 강점이 있지만 복잡성이 증가하고, 서버 측 디스커버리는 클라이언트의 단순성과 중앙 집중 관리에 강점이 있지만 추가 인프라와 오버헤드가 발생합니다. AWS 환경에서는 Cloud Map과 ECS Service Discovery를 활용하여 이 두 가지 방식의 장점을 결합하거나, 특정 요구사항에 맞춰 유연하게 서비스 디스커버리를 구현할 수 있습니다. 특히 Cloud Map은 다양한 AWS 리소스에 대한 통합된 디스커버리 솔루션을 제공하여 MSA 환경의 복잡성을 크게 줄여줍니다.
