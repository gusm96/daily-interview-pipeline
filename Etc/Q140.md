## 마이크로서비스 아키텍처(MSA)에서 API Gateway의 인증 및 인가 역할

### 1. 핵심 정의

API Gateway는 MSA에서 모든 클라이언트 요청의 단일 진입점 역할을 하는 서비스입니다. 단순한 요청 라우팅을 넘어, 보안, 모니터링, 로깅 등 다양한 횡단 관심사(Cross-cutting Concerns)를 처리하여 개별 마이크로서비스의 부담을 줄이고 아키텍처의 복잡성을 관리합니다.

### 2. 동작 원리 (인증 및 인가 관점)

API Gateway는 인증 및 인가 처리에서 다음과 같은 중요한 역할을 수행합니다.

*   **중앙 집중식 인증 처리:**
    *   클라이언트로부터 받은 인증 토큰(예: JWT, OAuth 토큰)을 검증하고, 유효한 경우 사용자 신원을 확인합니다.
    *   이를 통해 각 마이크로서비스는 자체적인 인증 로직을 구현할 필요 없이, 이미 인증된 요청만을 처리하게 됩니다.
    *   인증 방식 변경 시 API Gateway만 수정하면 되므로 유연성이 높습니다.
*   **중앙 집중식 인가 처리:**
    *   인증된 사용자의 역할(Role)이나 권한(Permission)을 기반으로 특정 API 엔드포인트에 대한 접근 허용 여부를 결정합니다.
    *   인가 정책을 API Gateway에 집중시켜 관리함으로써, 일관된 보안 정책을 적용하고 개별 서비스의 인가 로직 중복을 방지합니다.
    *   일반적으로 RBAC(Role-Based Access Control)나 ABAC(Attribute-Based Access Control)와 같은 모델을 적용할 수 있습니다.
*   **보안 강화:**
    *   악의적인 요청이나 비정상적인 접근 시도를 API Gateway 단에서 차단하여 내부 마이크로서비스를 보호합니다.
    *   Rate Limiting, IP Whitelisting/Blacklisting 등 추가적인 보안 기능을 통합하여 적용할 수 있습니다.

### 3. 실무에서 고려해야 할 사항

*   **성능 및 확장성:**
    *   모든 요청이 API Gateway를 통과하므로, 병목 현상이 발생하지 않도록 높은 성능과 수평적 확장성을 고려해야 합니다.
    *   캐싱 전략을 도입하여 반복적인 인증/인가 검증 부하를 줄일 수 있습니다.
*   **보안 강화:**
    *   API Gateway 자체가 단일 실패 지점(Single Point of Failure)이 될 수 있으므로, Gateway 자체의 보안 취약점 관리가 매우 중요합니다.
    *   TLS/SSL 적용, 민감 정보 로깅 최소화, 보안 패치 적용 등을 철저히 해야 합니다.
*   **정책 관리의 복잡성:**
    *   인증 및 인가 정책이 복잡해질수록 API Gateway의 설정 및 관리가 어려워질 수 있습니다.
    *   정책을 명확하게 정의하고, 버전 관리 및 테스트 전략을 수립하는 것이 중요합니다.
*   **장애 격리 및 복원력:**
    *   API Gateway에 장애가 발생하면 전체 시스템에 영향을 미칠 수 있으므로, 고가용성(High Availability) 및 장애 복구(Disaster Recovery) 전략을 수립해야 합니다.
    *   서킷 브레이커(Circuit Breaker) 패턴 등을 적용하여 특정 서비스의 장애가 Gateway로 전파되는 것을 방지할 수 있습니다.
*   **관찰 가능성(Observability):**
    *   API Gateway를 통해 들어오는 모든 요청에 대한 로깅, 모니터링, 트레이싱을 구현하여 시스템의 상태를 파악하고 문제 발생 시 신속하게 대응할 수 있도록 해야 합니다.
