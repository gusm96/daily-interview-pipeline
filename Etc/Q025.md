# [Q025] MSA API Gateway 역할과 구현

> 🧩 기타 (Python·FastAPI·Next.js / MSA·CI·CD·대용량·테스트) · 2026-06-29

**Q.** 마이크로서비스 아키텍처(MSA)에서 API Gateway는 중요한 역할을 수행합니다. API Gateway의 주요 기능(예: 라우팅, 인증/인가, 로드 밸런싱, 캐싱, 스로틀링)을 설명하고, MSA 환경에서 API Gateway를 도입했을 때 얻을 수 있는 이점과 발생할 수 있는 문제점에 대해 설명해주세요. 또한, AWS 환경에서 API Gateway를 활용하여 MSA를 구축하는 구체적인 방안을 설명해주세요.

## 🧑‍💻 나의 답변

## 마이크로서비스 아키텍처(MSA)에서 API Gateway

### 1. API Gateway의 주요 기능

API Gateway는 MSA에서 클라이언트와 마이크로서비스 간의 단일 진입점 역할을 하며, 다음과 같은 주요 기능을 수행합니다.

*   **라우팅 (Routing):** 클라이언트 요청을 적절한 마이크로서비스로 전달합니다. URL 경로, HTTP 메서드 등을 기반으로 요청을 특정 서비스 인스턴스로 라우팅합니다.
*   **인증/인가 (Authentication/Authorization):** 클라이언트의 신원을 확인(인증)하고, 해당 요청을 수행할 권한이 있는지 검증(인가)합니다. 토큰 검증, 사용자 권한 확인 등을 API Gateway에서 처리하여 각 서비스의 부담을 줄입니다.
*   **로드 밸런싱 (Load Balancing):** 여러 마이크로서비스 인스턴스에 요청을 분산하여 특정 서비스에 부하가 집중되는 것을 방지하고, 시스템의 가용성과 성능을 향상시킵니다.
*   **캐싱 (Caching):** 자주 요청되는 응답을 캐싱하여 마이크로서비스에 대한 불필요한 호출을 줄이고 응답 시간을 단축합니다.
*   **스로틀링 (Throttling):** 특정 클라이언트나 IP 주소로부터 오는 요청의 수를 제한하여 서비스 과부하를 방지하고, 서비스 안정성을 유지합니다.
*   **프로토콜 변환 (Protocol Translation):** 클라이언트와 서비스 간의 통신 프로토콜이 다를 경우 이를 변환해줍니다. (예: REST -> gRPC)
*   **모니터링 및 로깅 (Monitoring & Logging):** 모든 API 요청 및 응답에 대한 로그를 기록하고, 트래픽 및 성능 지표를 모니터링하여 시스템 운영에 필요한 정보를 제공합니다.

### 2. MSA 환경에서 API Gateway 도입 시 이점

*   **단일 진입점 제공:** 클라이언트는 여러 마이크로서비스의 엔드포인트를 알 필요 없이 API Gateway의 단일 엔드포인트만 호출하면 됩니다. 이는 클라이언트 개발을 단순화하고 서비스 변경에 대한 영향을 최소화합니다.
*   **서비스 추상화 및 캡슐화:** 내부 마이크로서비스의 복잡한 구조를 클라이언트로부터 숨겨줍니다. 서비스가 변경되거나 추가되어도 클라이언트는 API Gateway만 바라보면 됩니다.
*   **보안 강화:** 인증/인가, 스로틀링 등의 보안 기능을 중앙에서 관리하여 각 서비스의 보안 구현 부담을 줄이고 일관된 보안 정책을 적용할 수 있습니다.
*   **성능 최적화:** 캐싱, 로드 밸런싱 기능을 통해 시스템의 응답 속도를 향상시키고 부하를 분산하여 전반적인 성능을 최적화합니다.
*   **개발 및 배포 용이성:** 각 마이크로서비스는 독립적으로 개발 및 배포될 수 있으며, API Gateway는 이러한 변경 사항을 클라이언트에게 투명하게 제공합니다.
*   **모니터링 및 로깅 중앙화:** 모든 트래픽이 API Gateway를 통과하므로, 중앙 집중식으로 트래픽을 모니터링하고 로깅하여 문제 진단 및 분석을 용이하게 합니다.

### 3. MSA 환경에서 API Gateway 도입 시 문제점

*   **단일 실패 지점 (Single Point of Failure):** API Gateway 자체가 장애가 발생하면 전체 시스템에 접근할 수 없게 됩니다. 고가용성(HA) 설계를 통해 이를 방지해야 합니다.
*   **성능 병목 현상:** 모든 요청이 API Gateway를 통과하므로, API Gateway의 성능이 충분하지 않으면 병목 현상이 발생하여 전체 시스템의 성능 저하를 초래할 수 있습니다.
*   **복잡성 증가:** API Gateway 자체의 설정 및 관리가 복잡해질 수 있으며, 새로운 계층이 추가되면서 시스템 아키텍처의 복잡도가 증가합니다.
*   **추가적인 개발 및 운영 오버헤드:** API Gateway를 구축하고 유지보수하는 데 추가적인 개발 및 운영 리소스가 필요합니다.
*   **지연 시간 증가 (Latency):** 요청이 API Gateway를 한 번 더 거치게 되므로, 미미하지만 추가적인 네트워크 지연 시간이 발생할 수 있습니다.

### 4. AWS 환경에서 API Gateway를 활용한 MSA 구축 방안

AWS 환경에서 API Gateway를 활용하여 MSA를 구축하는 구체적인 방안은 다음과 같습니다.

1.  **AWS API Gateway 생성 및 설정:**
    *   **REST API 또는 HTTP API 선택:** 일반적으로 REST API는 더 많은 기능(예: 요청/응답 매핑 템플릿)을 제공하지만, HTTP API는 더 빠르고 비용 효율적입니다. MSA의 요구사항에 따라 적절한 API 유형을 선택합니다.
    *   **리소스 및 메서드 정의:** 각 마이크로서비스의 엔드포인트에 해당하는 리소스(예: `/users`, `/products`)와 HTTP 메서드(GET, POST 등)를 정의합니다.
    *   **통합 유형 설정:**
        *   **Lambda 통합:** AWS Lambda 함수로 구현된 마이크로서비스에 직접 연결합니다. 서버리스 MSA에 매우 적합합니다.
        *   **VPC Link 통합:** Amazon VPC 내부에 있는 ALB(Application Load Balancer) 또는 NLB(Network Load Balancer) 뒤에 있는 EC2 인스턴스, ECS/EKS 컨테이너 기반 마이크로서비스에 연결합니다.
        *   **HTTP/AWS 서비스 통합:** 외부 HTTP 엔드포인트나 다른 AWS 서비스(예: SQS, Kinesis)와 직접 통합할 수 있습니다.
    *   **인증/인가 설정:**
        *   **IAM 권한:** AWS IAM을 사용하여 API Gateway에 접근하는 사용자/역할에 대한 권한을 제어합니다.
        *   **Lambda Authorizer:** 커스텀 인증 로직을 Lambda 함수로 구현하여 API 요청을 인증/인가합니다. (예: JWT 토큰 검증)
        *   **Cognito User Pool Authorizer:** Amazon Cognito User Pool을 사용하여 사용자 인증을 처리합니다.
    *   **스로틀링 및 캐싱 설정:** API Gateway 단계(Stage) 설정에서 요청 제한(Rate Limiting) 및 버스트(Burst) 설정을 통해 스로틀링을 구성하고, 캐싱을 활성화하여 응답 시간을 단축합니다.
    *   **CORS 설정:** 클라이언트 웹 애플리케이션이 다른 도메인에서 API를 호출할 수 있도록 CORS(Cross-Origin Resource Sharing)를 설정합니다.

2.  **마이크로서비스 구현 및 배포:**
    *   **서버리스 마이크로서비스:** 각 마이크로서비스를 AWS Lambda 함수로 구현하고, API Gateway와 Lambda 통합을 통해 연결합니다. 이는 운영 오버헤드를 최소화하고 자동 스케일링의 이점을 얻을 수 있습니다.
    *   **컨테이너 기반 마이크로서비스:** Docker 컨테이너로 마이크로서비스를 구현하고, Amazon ECS(Elastic Container Service) 또는 Amazon EKS(Elastic Kubernetes Service)에 배포합니다. 이 경우, API Gateway는 VPC Link를 통해 ECS/EKS 클러스터 앞단의 ALB/NLB와 통합됩니다.
    *   **EC2 기반 마이크로서비스:** EC2 인스턴스에 직접 마이크로서비스를 배포하는 경우, ALB/NLB를 사용하여 로드 밸런싱을 구성하고 API Gateway와 VPC Link로 통합합니다.

3.  **모니터링 및 로깅:**
    *   **Amazon CloudWatch:** API Gateway는 CloudWatch와 자동으로 통합되어 API 호출 수, 지연 시간, 오류율 등의 지표를 모니터링하고, 상세한 액세스 로그를 CloudWatch Logs에 기록합니다. 이를 통해 시스템의 상태를 실시간으로 파악하고 문제 발생 시 빠르게 대응할 수 있습니다.
    *   **AWS X-Ray:** 분산 트레이싱을 위해 AWS X-Ray를 활성화하여 API Gateway를 통과하는 요청이 각 마이크로서비스를 거쳐가는 과정을 시각화하고, 성능 병목 지점을 식별합니다.

4.  **CI/CD 파이프라인 구축:**
    *   **AWS CodePipeline, CodeBuild, CodeDeploy:** 각 마이크로서비스와 API Gateway 설정을 위한 CI/CD 파이프라인을 구축하여 코드 변경 시 자동으로 빌드, 테스트, 배포가 이루어지도록 합니다. CloudFormation 또는 Serverless Framework와 같은 IaC(Infrastructure as Code) 도구를 사용하여 API Gateway 및 관련 리소스를 관리하면 배포의 일관성과 자동화를 높일 수 있습니다.

이러한 방안을 통해 AWS API Gateway는 MSA의 복잡성을 관리하고, 보안, 성능, 확장성을 향상시키는 핵심적인 역할을 수행할 수 있습니다.
