## HTTP/3, QUIC 프로토콜 및 성능 개선점

### 1. QUIC이 TCP 대신 UDP를 사용하는 이유

QUIC(Quick UDP Internet Connections)이 TCP 대신 UDP를 사용하는 주된 이유는 **TCP의 고질적인 문제점들을 회피하고, 더 유연하며 효율적인 전송 계층 프로토콜을 구현하기 위함**입니다.

*   **커널 레벨 구현의 한계:** TCP는 대부분 운영체제 커널 레벨에서 구현되어 있어, 새로운 기능 추가나 변경이 어렵습니다. 반면 UDP는 최소한의 기능만 제공하므로, QUIC은 사용자 공간(User-space)에서 자체적인 혼잡 제어, 신뢰성, 흐름 제어 등을 구현하여 애플리케이션 레벨에서 더 빠르게 진화하고 최적화될 수 있습니다.
*   **Head-of-Line Blocking (HOLB) 문제 회피:** TCP는 스트림 내에서 패킷 손실이 발생하면 해당 스트림의 모든 후속 패킷 처리가 중단됩니다. QUIC은 다중 스트림을 지원하며, 각 스트림은 독립적으로 동작하므로 한 스트림의 패킷 손실이 다른 스트림에 영향을 주지 않습니다.
*   **연결 설정 시간 단축 (0-RTT/1-RTT):** TCP는 3-way handshake를 통해 연결을 설정하고, TLS 암호화를 위해 추가적인 handshake가 필요하여 총 2-RTT가 소요됩니다. QUIC은 초기 연결 설정 시 암호화 및 전송 계층 핸드셰이크를 동시에 처리하여 1-RTT로 단축하며, 이전에 연결했던 서버의 정보를 캐싱하여 0-RTT로 연결을 재개할 수 있습니다.
*   **IP 주소 변경 시 연결 유지 (Connection Migration):** TCP는 연결이 4-tuple (Source IP, Source Port, Destination IP, Destination Port)로 식별되므로, 클라이언트의 IP 주소가 변경되면 기존 연결이 끊어지고 재설정해야 합니다. QUIC은 Connection ID를 사용하여 연결을 식별하므로, IP 주소나 포트가 변경되어도 동일한 Connection ID를 통해 연결을 끊김 없이 유지할 수 있습니다. 이는 모바일 환경에서 특히 유용합니다.

### 2. HTTP/3가 QUIC을 통해 얻는 주요 성능 개선점

HTTP/3는 QUIC 프로토콜을 기반으로 함으로써 다음과 같은 주요 성능 개선점을 얻습니다.

*   **Head-of-Line Blocking (HOLB) 해결:**
    *   **문제점:** HTTP/2는 하나의 TCP 연결 위에서 여러 개의 논리적인 스트림을 멀티플렉싱하지만, 하위 TCP 계층에서 패킷 손실이 발생하면 해당 TCP 연결의 모든 스트림이 멈추는 HOLB 문제가 발생합니다.
    *   **해결:** QUIC은 UDP 위에서 자체적인 스트림 멀티플렉싱을 구현합니다. 각 스트림은 독립적으로 신뢰성을 보장받으므로, 한 스트림에서 패킷 손실이 발생해도 다른 스트림의 데이터 전송에는 영향을 주지 않습니다. 이는 웹 페이지 로딩 시 여러 리소스(이미지, CSS, JS 등)를 동시에 요청할 때 특정 리소스의 손실이 전체 페이지 로딩을 지연시키는 것을 방지하여 체감 성능을 크게 향상시킵니다.

*   **연결 설정 시간 단축 (0-RTT/1-RTT):**
    *   **문제점:** HTTP/1.1 및 HTTP/2는 TCP 3-way handshake와 TLS handshake를 순차적으로 수행하여 최소 2-RTT의 연결 설정 시간이 필요합니다.
    *   **해결:** QUIC은 초기 연결 설정 시 전송 계층 핸드셰이크와 TLS 핸드셰이크를 통합하여 1-RTT로 단축합니다. 또한, 이전에 연결했던 서버에 대한 정보를 캐싱하여 재연결 시에는 클라이언트가 서버에 데이터를 보내는 동시에 연결을 설정하는 0-RTT 연결 재개를 지원합니다. 이는 특히 모바일 환경이나 지연 시간이 긴 네트워크에서 웹 페이지 로딩 시작 시간을 크게 단축시켜 사용자 경험을 개선합니다.

*   **연결 마이그레이션 (Connection Migration):**
    *   **문제점:** TCP 기반의 HTTP/1.1 및 HTTP/2는 클라이언트의 IP 주소나 포트가 변경되면 기존 연결이 끊어지고 새로운 연결을 다시 설정해야 합니다. 이는 Wi-Fi에서 LTE로 전환하거나, 네트워크 인터페이스가 변경될 때 발생합니다.
    *   **해결:** QUIC은 Connection ID를 사용하여 연결을 식별합니다. 클라이언트의 IP 주소나 포트가 변경되더라도 동일한 Connection ID를 사용하면 기존 연결을 끊김 없이 유지할 수 있습니다. 이는 모바일 환경에서 사용자가 네트워크 환경을 변경하더라도 웹 서비스의 연속성을 보장하여 끊김 없는 사용자 경험을 제공합니다.

이러한 개선점들을 통해 HTTP/3는 웹 성능을 전반적으로 향상시키고, 특히 모바일 및 불안정한 네트워크 환경에서 더욱 빠르고 안정적인 웹 경험을 제공합니다.
