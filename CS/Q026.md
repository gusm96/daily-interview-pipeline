## 운영체제(OS)에서 프로세스와 스레드의 주요 차이점, 동시성 문제 및 해결 기법

### 1. 프로세스와 스레드의 주요 차이점

| 특징         | 프로세스 (Process)                                   | 스레드 (Thread)                                        |
| :----------- | :--------------------------------------------------- | :----------------------------------------------------- |
| **정의**     | 실행 중인 프로그램의 인스턴스                        | 프로세스 내에서 실행되는 작업의 단위                   |
| **자원 소유** | 독립적인 메모리 공간(코드, 데이터, 힙, 스택) 소유   | 프로세스의 메모리 공간 및 자원 공유                    |
| **생성 비용** | 높은 비용 (독립적인 자원 할당)                       | 낮은 비용 (기존 프로세스 자원 활용)                    |
| **컨텍스트 스위칭** | 높은 비용 (메모리 맵 변경, 캐시 무효화 등)           | 낮은 비용 (동일한 주소 공간 내에서 레지스터만 변경)    |
| **통신**     | IPC(Inter-Process Communication) 필요 (파이프, 소켓 등) | 직접적인 메모리 접근을 통한 통신 (공유 변수 등)        |
| **독립성**   | 한 프로세스 오류가 다른 프로세스에 영향 미치지 않음 | 한 스레드 오류가 전체 프로세스에 영향 미칠 수 있음     |

**요약:** 프로세스는 운영체제로부터 독립적인 자원을 할당받아 실행되는 프로그램의 단위이며, 스레드는 이러한 프로세스 내에서 자원을 공유하며 실행되는 더 작은 실행 단위입니다. 스레드는 프로세스보다 생성 및 컨텍스트 스위칭 비용이 낮아 효율적인 동시성 처리에 유리합니다.

### 2. 멀티스레드 환경에서 발생할 수 있는 동시성 문제

멀티스레드 환경에서는 여러 스레드가 공유 자원에 동시에 접근하여 예상치 못한 결과를 초래할 수 있습니다.

*   **경쟁 조건 (Race Condition):** 여러 스레드가 공유 데이터에 동시에 접근하여 최종 결과가 어떤 스레드가 먼저 실행되었는지에 따라 달라지는 상황입니다. 예를 들어, 두 스레드가 동시에 카운터 변수를 증가시키려 할 때, 한 스레드의 업데이트가 다른 스레드의 업데이트를 덮어쓸 수 있습니다.
*   **교착 상태 (Deadlock):** 두 개 이상의 스레드가 서로가 가지고 있는 자원을 기다리며 무한히 대기하는 상태입니다. 예를 들어, 스레드 A는 자원 X를 점유하고 자원 Y를 기다리고, 스레드 B는 자원 Y를 점유하고 자원 X를 기다리는 상황입니다.
*   **기아 상태 (Starvation):** 특정 스레드가 필요한 자원을 계속해서 얻지 못하고 실행이 무한히 지연되는 상태입니다. 우선순위가 낮은 스레드가 계속해서 우선순위가 높은 스레드에 밀려 실행되지 못하는 경우가 대표적입니다.
*   **우선순위 역전 (Priority Inversion):** 우선순위가 높은 스레드가 우선순위가 낮은 스레드가 점유하고 있는 공유 자원을 기다리느라 실행이 지연되는 현상입니다.

### 3. 동시성 문제 해결을 위한 동기화 기법

동시성 문제를 해결하기 위해 스레드 간의 실행 순서나 공유 자원 접근을 제어하는 동기화 기법이 사용됩니다.

#### 3.1. 뮤텍스 (Mutex)

*   **동작 원리:** 상호 배제(Mutual Exclusion)를 의미하며, **하나의 스레드만이 공유 자원에 접근할 수 있도록** 합니다. 공유 자원에 접근하려는 스레드는 뮤텍스를 획득(lock)해야 하며, 사용이 끝나면 뮤텍스를 반환(unlock)합니다. 뮤텍스가 이미 획득된 상태라면 다른 스레드는 뮤텍스가 반환될 때까지 대기합니다.
*   **장점:**
    *   구현이 비교적 간단합니다.
    *   공유 자원에 대한 배타적인 접근을 보장하여 경쟁 조건을 효과적으로 방지합니다.
*   **단점:**
    *   세밀한 제어가 어렵습니다. (오직 1개의 스레드만 접근 가능)
    *   교착 상태가 발생할 수 있습니다.
    *   뮤텍스를 너무 많이 사용하면 성능 저하를 초래할 수 있습니다.
*   **비유:** 화장실 문에 "사용 중" 팻말을 걸어놓는 것과 같습니다. 한 명만 들어갈 수 있습니다.

#### 3.2. 세마포어 (Semaphore)

*   **동작 원리:** 공유 자원에 접근할 수 있는 스레드의 **개수를 제한**하는 카운터입니다. 세마포어는 정수 값을 가지며, `P()` (wait/acquire) 연산으로 값을 감소시키고, `V()` (signal/release) 연산으로 값을 증가시킵니다. 세마포어 값이 0이 되면 더 이상 자원에 접근할 수 없으며, 스레드는 대기합니다.
    *   **이진 세마포어 (Binary Semaphore):** 0 또는 1의 값을 가지며, 뮤텍스와 유사하게 동작합니다.
    *   **계수 세마포어 (Counting Semaphore):** 0 이상의 정수 값을 가지며, 여러 개의 동일한 자원에 대한 접근을 제어할 때 사용됩니다.
*   **장점:**
    *   뮤텍스보다 유연하게 여러 스레드의 공유 자원 접근을 제어할 수 있습니다.
    *   생산자-소비자 문제와 같은 복잡한 동기화 문제 해결에 유용합니다.
*   **단점:**
    *   구현이 복잡하고, 잘못 사용하면 교착 상태나 기아 상태를 유발할 수 있습니다.
    *   프로그래머의 실수로 `P()` 또는 `V()` 연산을 누락하면 문제가 발생하기 쉽습니다.
*   **비유:** 주차장에 빈 공간의 개수를 세는 것과 같습니다. 빈 공간이 있으면 주차할 수 있고, 없으면 기다려야 합니다.

#### 3.3. 모니터 (Monitor)

*   **동작 원리:** 공유 자원과 해당 자원에 접근하는 **메서드(프로시저)를 하나의 객체로 캡슐화**하여 동기화를 제공하는 고수준 동기화 기법입니다. 모니터 내의 메서드는 한 번에 하나의 스레드만 실행할 수 있도록 보장됩니다. 조건 변수(Condition Variable)를 사용하여 스레드 간의 복잡한 통신 및 대기를 처리할 수 있습니다.
*   **장점:**
    *   뮤텍스나 세마포어보다 추상화 수준이 높아 프로그래머가 동기화 로직을 직접 관리할 필요가 줄어듭니다.
    *   캡슐화를 통해 동기화 오류 발생 가능성을 낮춥니다.
    *   조건 변수를 통해 복잡한 스레드 간의 협력 및 대기 문제를 효과적으로 해결할 수 있습니다.
*   **단점:**
    *   언어 또는 런타임 환경에서 모니터를 지원해야 합니다. (Java의 `synchronized` 키워드, C#의 `lock` 키워드 등이 모니터 개념을 구현)
    *   세마포어와 마찬가지로 교착 상태가 발생할 수 있습니다.
*   **비유:** 은행 창구와 같습니다. 창구는 하나이고, 고객은 창구 직원의 안내에 따라 순서대로 업무를 처리합니다. 특정 조건(예: 대기표)이 충족될 때까지 기다릴 수도 있습니다.

### 각 기법 비교 요약

| 기법     | 주요 기능                                  | 제어 대상             | 추상화 수준 | 장점                                       | 단점                                       |
| :------- | :----------------------------------------- | :-------------------- | :---------- | :----------------------------------------- | :----------------------------------------- |
| **뮤텍스** | 상호 배제 (하나의 스레드만 접근)           | 공유 자원             | 낮음        | 간단한 구현, 경쟁 조건 방지                | 세밀한 제어 어려움, 교착 상태 가능성       |
| **세마포어** | 공유 자원 접근 개수 제한                 | 공유 자원 (여러 개)   | 중간        | 유연한 제어, 복잡한 동기화 문제 해결       | 복잡한 구현, 교착/기아 상태 가능성, 오류 발생 쉬움 |
| **모니터** | 공유 자원 및 메서드 캡슐화, 조건 변수 활용 | 공유 자원 및 관련 로직 | 높음        | 높은 추상화, 오류 감소, 복잡한 협력 처리   | 언어/환경 지원 필요, 교착 상태 가능성      |

이러한 동기화 기법들을 적절히 활용하여 멀티스레드 환경에서 발생할 수 있는 동시성 문제를 해결하고, 안정적이고 효율적인 시스템을 구축할 수 있습니다.
