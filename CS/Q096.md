# [Q096] OS 인터럽트 처리와 시스템 호출

### 모범 답안

운영체제에서 인터럽트와 시스템 호출은 모두 CPU의 제어권을 사용자 모드에서 커널 모드로 전환시키는 메커니즘이지만, 발생 원인과 처리 과정에서 차이가 있습니다.

#### 1. 인터럽트 (Interrupt)

*   **핵심 정의**: CPU가 프로그램을 실행하는 도중, 예기치 않은 상황이 발생하여 현재 실행 중인 작업을 중단하고 다른 작업을 처리하도록 하는 메커니즘입니다.
*   **동작 원리**:
    *   **발생 상황**: 주로 하드웨어 장치(I/O 완료, 타이머 만료 등)나 예외 상황(0으로 나누기, 잘못된 메모리 접근 등)에 의해 발생합니다.
    *   **처리 과정**:
        1.  인터럽트 발생 시, CPU는 현재 실행 중인 명령어의 주소와 레지스터 값 등 문맥 정보를 저장합니다.
        2.  미리 정의된 인터럽트 벡터 테이블(Interrupt Vector Table)을 참조하여 해당 인터럽트 서비스 루틴(ISR)의 시작 주소를 얻습니다.
        3.  ISR을 실행하여 인터럽트를 처리합니다. 이 과정은 커널 모드에서 진행됩니다.
        4.  ISR 완료 후, 저장했던 문맥 정보를 복원하고 중단되었던 사용자 프로그램으로 복귀합니다.
*   **실무에서 주의할 점**: 인터럽트 처리 루틴은 가능한 한 짧고 빠르게 동작해야 합니다. 너무 길어지면 다른 인터럽트나 중요한 작업의 지연을 초래하여 시스템 응답성에 악영향을 미칠 수 있습니다.

#### 2. 시스템 호출 (System Call)

*   **핵심 정의**: 사용자 프로그램이 운영체제의 커널 기능을 사용하기 위해 명시적으로 요청하는 메커니즘입니다.
*   **동작 원리**:
    *   **발생 상황**: 사용자 프로그램이 파일 입출력, 프로세스 생성/종료, 메모리 할당 등 운영체제의 자원이나 서비스를 필요로 할 때 발생합니다.
    *   **처리 과정**:
        1.  사용자 프로그램은 특정 시스템 호출 함수를 호출합니다. (예: `open()`, `read()`, `fork()`)
        2.  이 함수는 내부적으로 소프트웨어 인터럽트(트랩)를 발생시켜 CPU 제어권을 커널로 넘깁니다.
        3.  커널은 시스템 호출 번호를 확인하여 해당 시스템 호출 핸들러를 실행합니다. 이 과정은 커널 모드에서 진행됩니다.
        4.  시스템 호출 처리가 완료되면, 결과 값을 사용자 프로그램에 반환하고 제어권을 사용자 모드로 복귀시킵니다.
*   **실무에서 주의할 점**: 시스템 호출은 사용자 모드에서 커널 모드로의 전환 비용이 발생하므로, 불필요하게 많은 시스템 호출을 남발하면 성능 저하의 원인이 될 수 있습니다. 특히 루프 내에서 빈번한 시스템 호출은 피하는 것이 좋습니다.

#### 3. 시스템 성능에 미치는 영향

*   **모드 전환 오버헤드**: 인터럽트와 시스템 호출 모두 사용자 모드에서 커널 모드로의 전환과 복귀 과정에서 문맥 저장 및 복원 비용이 발생합니다. 이 오버헤드는 일반적으로 수십 나노초에서 수 마이크로초 단위로, 빈번하게 발생할 경우 전체 시스템 성능에 영향을 미칠 수 있습니다.
*   **자원 경합 및 동기화**: 커널 모드에서 공유 자원에 접근할 때, 여러 인터럽트나 시스템 호출이 동시에 발생하면 자원 경합 문제가 발생할 수 있습니다. 이를 해결하기 위한 락(lock) 등의 동기화 메커니즘은 추가적인 오버헤드를 발생시켜 성능에 영향을 줍니다.
*   **처리 시간**: 인터럽트 서비스 루틴이나 시스템 호출 핸들러의 처리 시간이 길어질수록, 해당 시간 동안 다른 작업들이 지연되므로 시스템의 응답성과 처리량에 부정적인 영향을 미칩니다.

따라서 운영체제는 이러한 메커니즘들을 효율적으로 설계하고 관리하여 성능 저하를 최소화하는 것이 중요합니다.
