## 애노테이션 프로세싱 (Annotation Processing)

### 1. 핵심 정의
애노테이션 프로세싱은 Java 컴파일러가 소스 코드를 컴파일하는 과정에서 특정 애노테이션을 찾아내고, 해당 애노테이션에 대한 정보를 기반으로 추가적인 코드 생성, 검증 또는 리소스 파일을 생성하는 메커니즘입니다. 이는 런타임 리플렉션과 달리 컴파일 시점에 동작하여 성능 오버헤드를 줄이고 타입 안정성을 확보하는 데 기여합니다.

### 2. 동작 원리
1. **컴파일러 플러그인**: 애노테이션 프로세서는 `javac` 컴파일러의 플러그인 형태로 동작합니다. 컴파일러는 소스 코드를 파싱하여 AST(Abstract Syntax Tree)를 생성하고, 이 과정에서 발견된 애노테이션들을 프로세서에게 전달합니다.
2. **`AbstractProcessor` 상속**: 애노테이션 프로세서는 `javax.annotation.processing.AbstractProcessor`를 상속받아 구현하며, `process()` 메서드에서 애노테이션 처리 로직을 정의합니다.
3. **`getSupportedAnnotationTypes()`**: 프로세서가 처리할 애노테이션 타입을 지정합니다.
4. **`getSupportedSourceVersion()`**: 프로세서가 지원하는 Java 소스 버전을 지정합니다.
5. **코드 생성/검증**: `process()` 메서드 내에서 `ProcessingEnvironment`를 통해 `Filer`를 사용하여 새로운 `.java` 소스 파일이나 리소스 파일을 생성하거나, `Messager`를 통해 컴파일 에러나 경고를 발생시켜 코드 검증을 수행할 수 있습니다. 생성된 코드는 다음 컴파일 라운드에 포함되어 함께 컴파일됩니다.

### 3. 실무에서 주의할 점 및 장단점

#### 장점
*   **성능 향상**: 런타임 리플렉션을 사용하는 대신 컴파일 시점에 코드를 생성하므로, 런타임 시 불필요한 오버헤드를 줄여 애플리케이션 성능을 향상시킬 수 있습니다.
    *   **예시**: Dagger, ButterKnife와 같은 DI(Dependency Injection) 라이브러리는 애노테이션 프로세싱을 통해 의존성 주입 코드를 미리 생성하여 런타임 리플렉션 사용을 최소화합니다.
*   **타입 안정성 확보**: 컴파일 시점에 코드 생성 및 검증이 이루어지므로, 런타임에 발생할 수 있는 오류를 미리 방지하고 타입 안정성을 높일 수 있습니다.
    *   **예시**: Lombok은 `@Getter`, `@Setter` 애노테이션을 통해 보일러플레이트 코드를 생성하여 개발자가 직접 작성하는 실수를 줄이고, 컴파일 시점에 유효성을 검증합니다.
*   **개발 생산성 향상**: 반복적이고 정형화된 코드를 자동으로 생성하여 개발자가 핵심 비즈니스 로직에 집중할 수 있도록 돕습니다.
    *   **예시**: MapStruct는 `@Mapper` 애노테이션을 사용하여 객체 간 매핑 코드를 자동으로 생성하여 수동 매핑 코드 작성의 번거로움을 줄입니다.

#### 한계점
*   **디버깅의 어려움**: 애노테이션 프로세서가 생성한 코드는 IDE에서 직접 확인하기 어렵거나, 디버깅 시 원본 소스 코드와 생성된 코드 사이의 흐름을 추적하기 어려울 수 있습니다.
*   **빌드 시간 증가**: 복잡하거나 많은 양의 코드를 생성하는 애노테이션 프로세서는 컴파일 시간을 증가시켜 빌드 성능에 영향을 줄 수 있습니다.
*   **학습 곡선**: 애노테이션 프로세서를 직접 구현하는 것은 Java 컴파일러 API에 대한 이해가 필요하며, 상대적으로 높은 학습 곡선을 가집니다.
*   **IDE 지원 문제**: 일부 IDE는 애노테이션 프로세서가 생성한 코드를 즉시 인식하지 못하여 빨간 줄이 표시되거나 자동 완성 기능이 제대로 동작하지 않는 경우가 발생할 수 있습니다. 일반적으로 IDE 재시작이나 빌드 도구(Gradle, Maven)를 통한 빌드 후에는 해결됩니다.
