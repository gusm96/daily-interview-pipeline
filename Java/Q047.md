## Java 불변(Immutable) 객체 설계 및 동시성 해결

### 1. 불변 객체 정의 및 설계 방법

**핵심 정의:**
불변 객체는 생성된 후 그 상태를 변경할 수 없는 객체입니다. 즉, 객체의 필드 값이 한 번 초기화되면 생명 주기 동안 절대 변하지 않습니다.

**설계 원리:**
불변 객체를 설계하기 위해서는 다음 규칙들을 따라야 합니다.

*   **모든 필드를 `private final`로 선언합니다.** 이를 통해 외부에서 직접 접근하거나 값을 변경하는 것을 막고, 한 번 초기화된 후에는 재할당을 방지합니다.
*   **Setter 메서드를 제공하지 않습니다.** 객체의 상태를 변경할 수 있는 어떠한 공용(public) 메서드도 제공하지 않아야 합니다.
*   **객체 생성 시 모든 필드를 초기화합니다.** 생성자를 통해 모든 필드의 값을 한 번에 설정해야 합니다.
*   **가변 객체를 참조하는 필드가 있다면 방어적 복사(Defensive Copy)를 수행합니다.** 생성자에서 가변 객체를 인자로 받을 때 내부 필드에 직접 할당하지 않고 새로운 객체를 생성하여 할당하며, Getter 메서드에서 가변 객체를 반환할 때도 원본 객체가 아닌 복사본을 반환해야 합니다.
*   **클래스를 `final`로 선언하여 상속을 방지합니다.** 상속을 통해 불변성을 깨뜨리는 서브클래스가 생성되는 것을 막습니다. (필수적인 것은 아니지만 강력히 권장됩니다.)

### 2. 불변 객체가 멀티스레드 환경에서 동시성 문제를 해결하는 방법

**동작 원리:**
불변 객체는 상태가 변하지 않으므로, 여러 스레드가 동시에 접근하더라도 데이터의 일관성이 깨질 염려가 없습니다.

*   **스레드 안전성:** 불변 객체는 생성 후 상태가 변경되지 않기 때문에, 어떤 스레드도 객체의 내부 상태를 변경할 수 없습니다. 따라서 별도의 동기화 메커니즘(락 등) 없이도 여러 스레드에서 안전하게 공유하고 사용할 수 있습니다. 이는 `race condition`이나 `deadlock`과 같은 동시성 문제로부터 자유롭다는 것을 의미합니다.
*   **가시성(Visibility):** 한 스레드가 불변 객체를 생성하면, 다른 스레드들은 항상 그 객체의 완전하고 일관된 상태를 볼 수 있습니다. 변경될 여지가 없으므로, 메모리 가시성 문제에 대한 걱정을 덜 수 있습니다.

### 3. 불변 객체 사용 시 주요 이점

*   **스레드 안전성:** 위에서 설명했듯이, 동기화 없이 여러 스레드에서 안전하게 공유할 수 있어 동시성 프로그래밍을 단순화합니다.
*   **캐싱 용이성:** 객체의 해시 코드가 한 번 계산되면 변경되지 않으므로, `HashMap`이나 `HashSet`과 같은 컬렉션의 키로 사용하기에 적합하며 캐싱에 유리합니다.
*   **오류 감소:** 객체의 상태가 예측 가능하고 변경되지 않으므로, 예상치 못한 부작용(side effect) 발생 가능성이 줄어들어 코드의 안정성과 디버깅 용이성이 향상됩니다.
*   **보안:** 객체의 상태가 외부에서 변경될 수 없으므로, 악의적인 코드나 실수로 인한 데이터 손상으로부터 객체를 보호할 수 있습니다.

### 4. 모든 객체를 불변으로 만들 수 없는 이유 및 트레이드오프

**모든 객체를 불변으로 만들 수 없는 이유:**

*   **성능 오버헤드:** 객체의 상태가 변경될 때마다 새로운 객체를 생성해야 하므로, 빈번한 상태 변경이 필요한 경우 가비지 컬렉션 부하가 증가하고 성능 저하를 초래할 수 있습니다. 예를 들어, 문자열을 조작하는 `String` 클래스는 불변이지만, 반복적인 문자열 조작에는 `StringBuilder`나 `StringBuffer`와 같은 가변 객체를 사용하는 것이 효율적입니다.
*   **메모리 사용량 증가:** 상태가 변경될 때마다 새로운 객체가 생성되므로, 많은 수의 객체가 생성되고 소멸되어 메모리 사용량이 증가할 수 있습니다.

**트레이드오프:**

불변 객체는 스레드 안전성, 예측 가능성, 안정성 등 많은 이점을 제공하지만, 성능 및 메모리 사용량 측면에서 트레이드오프가 존재합니다. 따라서 객체의 특성과 사용 패턴을 고려하여 불변성을 적용할지 결정해야 합니다.

*   **불변 객체가 적합한 경우:** 상태 변경이 적거나, 여러 스레드에서 공유될 가능성이 높은 객체(예: 설정 객체, DTO, 값 객체)
*   **가변 객체가 적합한 경우:** 상태 변경이 빈번하여 새로운 객체 생성이 비효율적인 경우(예: 빌더 패턴의 중간 객체, 대용량 데이터 처리 객체)

실무에서는 이 두 가지 접근 방식을 적절히 조합하여 사용하며, 불변 객체의 장점을 최대한 활용하면서도 성능 저하를 최소화하는 것이 중요합니다.
