## Java 가비지 컬렉션(GC) 동작 원리 및 튜닝 전략

Java의 가비지 컬렉션(GC)은 JVM의 힙(Heap) 영역에서 더 이상 참조되지 않는 객체를 자동으로 회수하여 메모리 누수를 방지하고 개발 편의성을 높이는 핵심 기능입니다. 하지만 GC의 동작 방식을 이해하지 못하면 불필요한 GC 발생으로 인해 애플리케이션 성능 저하를 초래할 수 있습니다.

### 1. JVM 힙(Heap) 영역 구조 및 GC 동작 원리

JVM의 힙 영역은 효율적인 GC를 위해 여러 세대(Generation)로 나뉘어 있습니다. Oracle Java Reference에 따르면, 대부분의 객체는 생성된 후 짧은 시간 내에 더 이상 사용되지 않는다는 "약한 세대 가설(Weak Generational Hypothesis)"에 기반합니다.

*   **Young Generation (젊은 세대)**
    *   새롭게 생성된 객체들이 처음 할당되는 공간입니다.
    *   **Eden Space**: 대부분의 새 객체가 처음 할당되는 공간입니다.
    *   **Survivor Space (S0, S1)**: Eden Space에서 살아남은 객체들이 이동하는 공간으로, 두 개의 공간이 번갈아 가며 사용됩니다.
    *   **Minor GC (Young GC)**: Young Generation에서 발생하는 GC입니다. Eden Space가 가득 차면 발생하며, 살아남은 객체들은 Survivor Space로 이동하거나, 특정 임계값(age)을 넘으면 Old Generation으로 승격(Promotion)됩니다. Minor GC는 일반적으로 매우 빠르고 자주 발생합니다.

*   **Old Generation (오래된 세대)**
    *   Young Generation에서 오랫동안 살아남아 승격된 객체들이 저장되는 공간입니다.
    *   **Major GC (Old GC) / Full GC**: Old Generation에서 발생하는 GC입니다. Old Generation이 가득 차면 발생하며, Young Generation과 Old Generation 전체를 대상으로 합니다. Major GC는 Minor GC보다 훨씬 많은 시간을 소요하며, 애플리케이션의 일시 정지(Stop-The-World, STW) 시간을 길게 만들 수 있어 성능에 큰 영향을 미칩니다.

*   **Permanent Generation (Java 8 이전) / Metaspace (Java 8 이후)**
    *   클래스 메타데이터, 메서드 정보, 상수 풀 등 JVM이 사용하는 데이터를 저장하는 공간입니다.
    *   Java 8부터는 Metaspace로 변경되어 Native Memory를 사용하며, `MaxMetaspaceSize`로 크기를 제한할 수 있습니다. 이 영역의 GC는 Full GC에 포함되거나 별도로 발생할 수 있습니다.

### 2. GC 튜닝 시 고려해야 할 주요 지표

GC 튜닝은 애플리케이션의 특성과 요구사항에 맞춰 최적의 GC 동작을 찾아가는 과정입니다. 다음 지표들을 모니터링하며 튜닝 방향을 결정합니다.

*   **GC 발생 빈도 (Frequency)**: GC가 얼마나 자주 발생하는지 나타냅니다. 너무 잦은 GC는 불필요한 오버헤드를 발생시킵니다.
*   **GC 소요 시간 (Duration)**: 한 번의 GC가 완료되는 데 걸리는 시간입니다. 특히 Major GC/Full GC의 소요 시간은 STW로 이어져 사용자 경험에 직접적인 영향을 미칩니다.
*   **GC 처리량 (Throughput)**: GC에 소요된 시간을 제외하고 애플리케이션이 작업을 수행하는 데 사용된 시간의 비율입니다. 높은 처리량은 GC로 인한 성능 저하가 적음을 의미합니다.
*   **GC 일시 정지 시간 (Pause Time)**: GC가 애플리케이션 스레드를 멈추는 STW 시간입니다. 짧은 일시 정지 시간은 실시간 서비스에 중요합니다.
*   **힙 사용량 (Heap Usage)**: 힙 메모리가 얼마나 사용되고 있는지, GC 후에도 충분한 여유 공간이 확보되는지 확인합니다.
*   **승격률 (Promotion Rate)**: Young Generation에서 Old Generation으로 객체가 승격되는 비율입니다. 너무 높은 승격률은 Old Generation의 빠른 소진과 Major GC의 빈번한 발생으로 이어질 수 있습니다.

### 3. 일반적인 GC 튜닝 전략

GC 튜닝은 주로 JVM 옵션을 통해 이루어지며, 목표는 애플리케이션의 요구사항(처리량, 응답 시간 등)에 맞춰 GC 오버헤드를 최소화하는 것입니다.

*   **힙 메모리 크기 조정 (`-Xms`, `-Xmx`)**:
    *   **전략**: 애플리케이션이 필요로 하는 최소/최대 힙 크기를 적절히 설정합니다. 너무 작으면 잦은 GC가 발생하고, 너무 크면 GC 한 번에 소요되는 시간이 길어질 수 있습니다. 일반적으로 `-Xms`와 `-Xmx`를 동일하게 설정하여 힙 크기 조절로 인한 오버헤드를 줄입니다.
    *   **고려사항**: 서버의 물리적 메모리 한계와 다른 프로세스들의 메모리 사용량을 고려해야 합니다.

*   **Young Generation 크기 조정 (`-Xmn` 또는 `-XX:NewRatio`)**:
    *   **전략**: Young Generation의 크기를 조절하여 Minor GC의 빈도와 효율성을 제어합니다.
        *   `Young Generation`을 크게 하면 Minor GC 빈도는 줄어들지만, 한 번의 Minor GC 소요 시간은 길어질 수 있습니다.
        *   `Young Generation`을 작게 하면 Minor GC 빈도는 늘어나지만, 한 번의 Minor GC 소요 시간은 짧아집니다.
    *   **고려사항**: 대부분의 객체가 빠르게 죽는다면 Young Generation을 충분히 크게 하여 Old Generation으로의 승격을 줄이는 것이 좋습니다.

*   **GC 알고리즘 선택 (`-XX:+UseG1GC`, `-XX:+UseParallelGC`, `-XX:+UseConcMarkSweepGC` 등)**:
    *   **전략**: Java 8 기본 GC는 Parallel GC였으나, Java 9부터 G1 GC가 기본 GC로 변경되었습니다. 각 GC 알고리즘은 다른 특성을 가집니다.
        *   **Parallel GC**: 높은 처리량(Throughput)을 목표로 하며, 멀티 코어 CPU 환경에서 Minor GC와 Major GC를 병렬로 처리합니다. STW 시간이 길 수 있습니다.
        *   **CMS GC (Concurrent Mark Sweep GC)**: 낮은 일시 정지 시간(Low Pause Time)을 목표로 하며, 대부분의 작업을 애플리케이션 스레드와 동시에 수행합니다. 하지만 단편화(Fragmentation) 문제가 발생할 수 있고, Java 9부터 Deprecated 되었습니다.
        *   **G1 GC (Garbage First GC)**: 대용량 힙(4GB 이상)에서 낮은 일시 정지 시간과 높은 처리량을 모두 목표로 합니다. 힙을 Region으로 나누어 관리하며, 가장 효율적인 Region부터 GC를 수행합니다.
        *   **ZGC / Shenandoah GC (Java 11+)**: 매우 낮은 일시 정지 시간(10ms 미만)을 목표로 하는 최신 GC입니다. 대용량 힙에서도 거의 일정한 STW 시간을 보장합니다.
    *   **고려사항**: 애플리케이션의 요구사항(처리량 vs 응답 시간)과 힙 크기에 따라 적절한 GC를 선택합니다.

*   **객체 승격 임계값 조정 (`-XX:MaxTenuringThreshold`)**:
    *   **전략**: Young Generation에서 객체가 Old Generation으로 승격되기까지 Minor GC를 몇 번 견뎌야 하는지 설정합니다.
    *   **고려사항**: 이 값을 너무 낮게 설정하면 Young Generation에서 충분히 죽을 수 있는 객체들이 Old Generation으로 불필요하게 승격되어 Major GC를 유발할 수 있습니다.

*   **GC 로깅 활성화 (`-Xlog:gc*`)**:
    *   **전략**: GC 튜닝의 시작은 현재 GC 동작을 정확히 파악하는 것입니다. GC 로그를 활성화하여 GC 발생 시간, 소요 시간, 힙 사용량 등을 상세히 기록하고 분석합니다.
    *   **도구**: GCViewer, GCEasy 등 GC 로그 분석 도구를 활용하여 시각적으로 GC 동작을 이해하고 문제점을 파악할 수 있습니다.

**결론적으로,** Java GC 튜닝은 애플리케이션의 특성과 요구사항을 면밀히 분석하고, GC 지표를 지속적으로 모니터링하며, 적절한 JVM 옵션과 GC 알고리즘을 선택하여 최적의 성능을 찾아가는 반복적인 과정입니다. Oracle Java Reference를 참고하여 각 GC 알고리즘의 상세한 동작 방식과 옵션을 이해하는 것이 중요합니다.
