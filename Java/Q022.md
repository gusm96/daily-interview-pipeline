## Java 제네릭(Generics)의 타입 소거(Type Erasure)와 한계점, 그리고 극복 방법

### 1. 타입 소거(Type Erasure) 개념

Java 제네릭은 컴파일 시점에 타입 안정성을 제공하여 잘못된 타입의 객체가 컬렉션에 추가되는 것을 방지합니다. 하지만 런타임에는 **타입 소거(Type Erasure)**가 발생합니다. 이는 제네릭 타입 정보가 런타임에 제거되고, 원시 타입(raw type)으로 대체되는 것을 의미합니다. 예를 들어, `List<String>`은 런타임에 `List`로 변환됩니다.

**Oracle Java Reference 기반 설명:**
"The Java compiler applies type erasure to:
* Replace all type parameters in generic types with their bounds or `Object` if the type parameters are unbounded.
* Insert type casts to preserve type safety.
* Generate bridge methods to preserve polymorphism in extended generic types."
(출처: Oracle Java Documentation - Type Erasure)

즉, 컴파일러는 제네릭 타입 파라미터를 해당 바운드(bound) 또는 `Object`로 대체하고, 타입 안전성을 유지하기 위한 형변환(cast)을 삽입하며, 제네릭 타입을 확장할 때 다형성을 보존하기 위한 브릿지 메서드를 생성합니다.

### 2. 타입 소거로 인한 제네릭의 한계점

타입 소거는 다음과 같은 한계점을 발생시킵니다.

*   **런타임에 타입 정보 소실:** 런타임에는 제네릭 타입 인자 정보가 사라지므로, `List<String>`과 `List<Integer>`는 런타임에 모두 `List`로 인식됩니다.
    ```java
    List<String> stringList = new ArrayList<>();
    List<Integer> integerList = new ArrayList<>();

    System.out.println(stringList.getClass() == integerList.getClass()); // true
    // 런타임에는 두 객체 모두 ArrayList 클래스로 인식됩니다.
    ```

*   **`instanceof` 연산자 사용 불가:** 런타임에 제네릭 타입 인자 정보가 없으므로, `instanceof` 연산자를 제네릭 타입과 함께 사용할 수 없습니다.
    ```java
    // if (obj instanceof List<String>) { ... } // 컴파일 오류 발생
    ```
    이는 런타임에 `List<String>`이 `List`로 소거되기 때문에, `instanceof` 연산자가 특정 제네릭 타입 인자를 확인할 수 없기 때문입니다.

*   **제네릭 타입의 배열 생성 불가:** `new T[size]`와 같이 제네릭 타입의 배열을 직접 생성할 수 없습니다.
    ```java
    // T[] array = new T[10]; // 컴파일 오류 발생
    ```
    이는 배열은 런타임에 자신의 컴포넌트 타입 정보를 알아야 하는데, 제네릭 타입은 런타임에 타입 정보가 소거되기 때문입니다.

*   **제네릭 타입 파라미터로 오버로드 불가:** 제네릭 타입 파라미터만 다른 메서드를 오버로드할 수 없습니다.
    ```java
    // void print(List<String> list) { ... }
    // void print(List<Integer> list) { ... } // 컴파일 오류 발생 (동일한 시그니처로 간주)
    ```
    런타임에는 두 메서드 모두 `print(List list)`로 소거되므로, 컴파일러는 이를 동일한 시그니처로 간주하여 충돌을 일으킵니다.

### 3. 한계점을 극복하기 위한 방법

타입 소거로 인한 한계를 극복하기 위해 다음과 같은 방법들이 사용됩니다.

*   **타입 토큰 (Type Token):**
    타입 토큰은 런타임에 제네릭 타입 정보를 유지하기 위한 일반적인 패턴입니다. `Class<T>` 객체를 메서드 인자로 전달하여 런타임에 필요한 타입 정보를 제공합니다.

    ```java
    public <T> T createInstance(Class<T> type) throws InstantiationException, IllegalAccessException {
        return type.newInstance();
    }

    // 사용 예시
    String str = createInstance(String.class);
    Integer num = createInstance(Integer.class);
    ```
    이 방법은 `Class<T>` 자체가 제네릭 타입 `T`에 대한 런타임 정보를 가지고 있기 때문에 가능합니다.

*   **슈퍼 타입 토큰 (Super Type Token):**
    타입 토큰의 한계는 `List<String>`과 같은 복합 제네릭 타입의 정보를 얻기 어렵다는 점입니다. 이를 해결하기 위해 **슈퍼 타입 토큰** 패턴이 사용됩니다. 이는 익명 내부 클래스의 제네릭 슈퍼 타입을 활용하여 런타임에 제네릭 타입 인자 정보를 추출하는 방법입니다.

    **Oracle Java Reference 기반 설명:**
    "The Java Language Specification (JLS) states that the actual type arguments of a parameterized type are preserved at runtime if the parameterized type is a direct supertype of an anonymous class."
    (출처: Oracle Java Documentation - Type Erasure and Reification)

    즉, 익명 클래스의 직접적인 슈퍼 타입인 경우, 매개변수화된 타입의 실제 타입 인수는 런타임에 보존됩니다.

    ```java
    import java.lang.reflect.ParameterizedType;
    import java.lang.reflect.Type;

    public abstract class TypeReference<T> {
        private final Type type;

        protected TypeReference() {
            Type superclass = getClass().getGenericSuperclass();
            if (superclass instanceof Class) {
                throw new RuntimeException("Missing type parameter.");
            }
            // TypeReference<T>에서 T의 실제 타입을 추출
            this.type = ((ParameterizedType) superclass).getActualTypeArguments()[0];
        }

        public Type getType() {
            return this.type;
        }
    }

    // 사용 예시
    Type stringListType = new TypeReference<List<String>>() {}.getType();
    System.out.println(stringListType); // java.util.List<java.lang.String>
    ```
    `TypeReference`의 익명 서브클래스를 생성함으로써, `TypeReference<List<String>>`의 `List<String>` 정보가 런타임에 보존되고, 이를 `getGenericSuperclass()`와 `getActualTypeArguments()`를 통해 추출할 수 있습니다. 이는 주로 JSON 직렬화/역직렬화 라이브러리(예: Jackson, Gson)에서 복잡한 제네릭 타입의 객체를 처리할 때 활용됩니다.

이러한 방법들을 통해 Java 제네릭의 타입 소거로 인한 런타임 한계를 극복하고, 더욱 유연하고 타입 안전한 코드를 작성할 수 있습니다.
