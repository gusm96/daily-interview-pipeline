## Java 메모리 모델(JMM)과 멀티스레드 환경에서의 가시성 및 순서 보장

Java 메모리 모델(JMM)은 멀티스레드 환경에서 변수의 가시성(Visibility)과 순서(Ordering)를 보장하기 위한 규칙 집합입니다. JMM은 스레드가 메인 메모리에서 값을 읽고 쓰는 방식, 그리고 컴파일러 및 프로세서가 코드 실행 순서를 재배치하는 방식에 대한 제약을 정의하여, 스레드 간의 일관된 메모리 뷰를 제공합니다.

### 가시성(Visibility) 보장

가시성은 한 스레드가 변경한 변수의 값이 다른 스레드에게 언제 보이는지를 의미합니다. 각 스레드는 자체적인 로컬 캐시를 가질 수 있으므로, 한 스레드가 변수를 변경하더라도 해당 변경 사항이 즉시 메인 메모리에 반영되지 않거나, 다른 스레드의 로컬 캐시가 무효화되지 않으면 다른 스레드는 오래된 값을 볼 수 있습니다.

### 순서(Ordering) 보장

순서는 코드에 작성된 명령문의 실행 순서가 실제로 메모리에서 관찰되는 순서와 일치하는지를 의미합니다. 컴파일러나 프로세서는 성능 최적화를 위해 명령문의 순서를 재배치(reordering)할 수 있습니다. 이러한 재배치는 단일 스레드 환경에서는 문제가 없지만, 멀티스레드 환경에서는 예상치 못한 동작을 유발할 수 있습니다.

### `volatile` 키워드와 `synchronized` 키워드의 역할 (JMM 관점)

Oracle Java Reference에 따르면, `volatile`과 `synchronized` 키워드는 JMM 관점에서 다음과 같은 역할을 하며, 가시성과 순서를 보장합니다.

#### 1. `volatile` 키워드

`volatile` 키워드는 변수에 적용되며, 주로 **가시성**을 보장하는 데 사용됩니다.

*   **역할:**
    *   **쓰기 작업:** `volatile` 변수에 대한 쓰기 작업은 해당 스레드의 로컬 캐시를 무효화하고, 변경된 값을 즉시 메인 메모리에 기록하도록 강제합니다. 이는 "Write Barrier"와 유사하게 작동하여, 해당 쓰기 작업 이전에 발생한 모든 메모리 쓰기 작업이 메인 메모리에 반영되도록 보장합니다.
    *   **읽기 작업:** `volatile` 변수에 대한 읽기 작업은 항상 메인 메모리에서 최신 값을 읽어오도록 강제합니다. 이는 "Read Barrier"와 유사하게 작동하여, 해당 읽기 작업 이후에 발생하는 모든 메모리 읽기 작업이 최신 값을 기반으로 하도록 보장합니다.
    *   **순서 보장 (부분적):** `volatile` 변수에 대한 읽기/쓰기 작업은 그 자체로 메모리 배리어(Memory Barrier) 역할을 합니다. 즉, `volatile` 쓰기 이전의 모든 메모리 작업은 `volatile` 쓰기 이전에 완료되어야 하며, `volatile` 읽기 이후의 모든 메모리 작업은 `volatile` 읽기 이후에 시작되어야 합니다. 이는 `volatile` 변수 자체에 대한 재배치를 방지하고, `volatile` 변수를 기준으로 다른 메모리 작업의 순서가 보장되도록 합니다.

*   **사용 상황:**
    *   **단순 플래그(Flag) 변수:** 한 스레드가 값을 변경하고 다른 스레드가 그 값을 읽어 특정 조건을 확인하는 경우 (예: `boolean shutdownRequested = false;`).
    *   **상태 변수:** 여러 스레드에서 공유되지만, 원자적인 연산(단일 읽기/쓰기)만 필요한 경우.
    *   **Double-Checked Locking 패턴:** 인스턴스 초기화 시 `volatile`을 사용하여 부분적으로 초기화된 객체가 다른 스레드에 노출되는 것을 방지합니다.

*   **제한 사항:** `volatile`은 원자성(Atomicity)을 보장하지 않습니다. 즉, `i++`와 같은 복합 연산은 `volatile`로 선언하더라도 스레드 안전하지 않습니다. 이는 `i++`가 읽기, 증가, 쓰기의 세 단계로 이루어지기 때문입니다.

#### 2. `synchronized` 키워드

`synchronized` 키워드는 메서드나 코드 블록에 적용되며, **가시성**과 **순서**는 물론 **원자성(Atomicity)**까지 보장합니다.

*   **역할:**
    *   **상호 배제 (Mutual Exclusion):** `synchronized` 블록이나 메서드는 한 번에 하나의 스레드만 실행할 수 있도록 보장합니다. 이는 락(Lock) 메커니즘을 통해 이루어집니다. 스레드가 `synchronized` 블록에 진입하려면 해당 락을 획득해야 하고, 블록을 벗어날 때 락을 해제합니다.
    *   **가시성 보장:**
        *   **락 획득 시:** `synchronized` 블록에 진입하는 스레드는 해당 락과 관련된 모든 변수의 최신 값을 메인 메모리에서 읽어와 로컬 캐시를 동기화합니다. (Acquire Fence)
        *   **락 해제 시:** `synchronized` 블록을 벗어나는 스레드는 해당 락과 관련된 모든 변경된 변수 값을 메인 메모리에 기록하고, 다른 스레드의 로컬 캐시를 무효화합니다. (Release Fence)
    *   **순서 보장:** 락 획득 및 해제 시점은 강력한 메모리 배리어 역할을 합니다. `synchronized` 블록 내의 모든 메모리 작업은 블록 진입 시점 이후에 발생하고, 블록 탈출 시점 이전에 완료되어야 합니다. 이는 컴파일러 및 프로세서의 재배치를 제한하여, `synchronized` 블록 내의 코드 실행 순서가 예측 가능하도록 합니다.

*   **사용 상황:**
    *   **공유 데이터의 일관성 유지:** 여러 스레드가 동시에 접근하여 변경할 수 있는 공유 데이터 구조(예: `ArrayList`를 스레드 안전하게 사용해야 할 때)를 보호할 때.
    *   **복합 연산의 원자성 보장:** `i++`와 같이 여러 단계로 이루어진 연산을 스레드 안전하게 수행해야 할 때.
    *   **임계 영역(Critical Section) 보호:** 특정 코드 블록이 한 번에 하나의 스레드에 의해서만 실행되어야 할 때.

### `volatile`과 `synchronized` 비교 요약

| 특징         | `volatile`                                   | `synchronized`                               |
| :----------- | :------------------------------------------- | :------------------------------------------- |
| **적용 대상**  | 변수                                         | 메서드 또는 코드 블록                        |
| **주요 보장** | 가시성, 부분적인 순서 보장                   | 가시성, 순서 보장, 원자성, 상호 배제         |
| **성능**     | `synchronized`보다 가볍고 빠름               | `volatile`보다 오버헤드가 큼                 |
| **사용 목적**  | 단순 플래그, 상태 변수, 단일 쓰기/읽기       | 공유 데이터 보호, 복합 연산, 임계 영역       |
| **내부 동작**  | 메모리 배리어(Read/Write Barrier)            | 락(Lock) 메커니즘 (Acquire/Release Fence)    |
| **원자성**   | 보장하지 않음                                | 보장함                                       |

### 결론

JMM은 멀티스레드 환경에서 데이터 일관성을 유지하기 위한 핵심적인 개념입니다. `volatile` 키워드는 주로 단일 변수의 가시성을 보장하고 부분적인 순서 제약을 제공하여, 단순한 상태 플래그나 단일 쓰기/읽기 연산에 적합합니다. 반면, `synchronized` 키워드는 락 메커니즘을 통해 가시성, 순서, 그리고 원자성까지 포괄적으로 보장하여, 복잡한 공유 데이터 구조나 복합 연산을 보호하는 데 사용됩니다. 올바른 키워드를 적절한 상황에 사용하는 것은 스레드 안전하고 성능 좋은 애플리케이션을 개발하는 데 필수적입니다.
