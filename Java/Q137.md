## Java 동시성 문제 및 스레드 안전성 기법

면접관님, Java에서 여러 스레드가 공유 자원에 동시에 접근할 때 발생할 수 있는 대표적인 동시성 문제들과 이를 해결하기 위한 스레드 안전성 기법에 대해 설명드리겠습니다.

### 1. 대표적인 동시성 문제

여러 스레드가 공유 자원에 동시에 접근할 때 발생할 수 있는 주요 동시성 문제들은 다음과 같습니다.

*   **경쟁 조건 (Race Condition):** 여러 스레드가 공유 자원에 접근하여 최종 결과가 스레드 실행 순서에 따라 달라지는 문제입니다. 이는 주로 읽기-수정-쓰기(Read-Modify-Write) 작업에서 발생하며, 예상치 못한 결과를 초래합니다.
*   **교착 상태 (Deadlock):** 두 개 이상의 스레드가 서로가 가지고 있는 자원을 기다리며 무한히 대기하는 상태입니다. 각 스레드가 필요한 자원을 얻지 못해 더 이상 진행할 수 없게 됩니다.
*   **기아 상태 (Starvation):** 특정 스레드가 필요한 자원을 계속해서 얻지 못하거나, 우선순위가 높은 스레드에 밀려 오랫동안 실행되지 못하는 상태입니다.
*   **라이브락 (Livelock):** 두 개 이상의 스레드가 서로의 작업을 방해하지 않으려고 계속해서 상태를 변경하지만, 결국 아무런 유의미한 작업도 진행하지 못하는 상태입니다. 교착 상태와 유사하지만, 스레드들이 멈춰있지 않고 계속해서 활동한다는 차이가 있습니다.

### 2. 스레드 안전성(Thread Safety) 기법

Java는 이러한 동시성 문제를 해결하고 스레드 안전성을 확보하기 위해 다양한 기법을 제공합니다.

#### 2.1. `synchronized` 키워드

*   **핵심 정의:** 특정 코드 블록이나 메서드를 한 번에 하나의 스레드만 실행할 수 있도록 보장하는 상호 배제(Mutual Exclusion) 메커니즘입니다. 내부적으로 모니터(Monitor) 또는 내장 락(Intrinsic Lock)을 사용합니다.
*   **동작 원리:** 스레드가 `synchronized` 블록이나 메서드에 진입하기 전에 해당 객체의 락을 획득하고, 블록을 벗어나면 락을 해제합니다. 다른 스레드는 락이 해제될 때까지 대기합니다.
*   **장점:**
    *   사용법이 간단하고 직관적입니다.
    *   JVM에 의해 관리되므로 개발자가 직접 락을 해제하는 것을 잊을 염려가 적습니다.
    *   메모리 가시성(Memory Visibility)을 보장합니다 (락 획득 시 캐시 무효화, 락 해제 시 캐시 쓰기).
*   **단점:**
    *   락을 획득하지 못한 스레드는 블로킹(Blocking)되어 성능 저하를 유발할 수 있습니다.
    *   세밀한 락 제어가 어렵고, 락의 범위가 넓어지면 병렬성을 저해할 수 있습니다.
    *   교착 상태 발생 가능성이 있습니다.

#### 2.2. `java.util.concurrent.locks` 패키지 (ReentrantLock 등)

*   **핵심 정의:** `synchronized` 키워드보다 유연하고 강력한 락 기능을 제공하는 인터페이스와 클래스 집합입니다. 대표적으로 `ReentrantLock`이 있습니다.
*   **동작 원리:** 개발자가 명시적으로 `lock()`과 `unlock()` 메서드를 호출하여 락을 획득하고 해제합니다. 조건 변수(`Condition`)와 함께 사용하여 더 복잡한 동기화 시나리오를 구현할 수 있습니다.
*   **장점:**
    *   `tryLock()`을 통한 비블로킹 락 획득 시도, `lockInterruptibly()`를 통한 인터럽트 가능한 락 획득 등 유연한 락 제어가 가능합니다.
    *   읽기-쓰기 락(`ReentrantReadWriteLock`)과 같이 특정 상황에 최적화된 락을 제공하여 병렬성을 높일 수 있습니다.
    *   여러 개의 조건 변수를 사용하여 복잡한 스레드 간 통신을 구현할 수 있습니다.
*   **단점:**
    *   개발자가 직접 `lock()`과 `unlock()`을 관리해야 하므로, `unlock()` 호출을 누락할 경우 교착 상태나 자원 누수가 발생할 위험이 있습니다. (일반적으로 `finally` 블록에서 `unlock()`을 호출하여 방지)
    *   `synchronized`에 비해 사용법이 복잡합니다.

#### 2.3. `java.util.concurrent.atomic` 패키지 (AtomicInteger 등)

*   **핵심 정의:** 단일 변수에 대한 원자적(Atomic) 연산을 제공하는 클래스 집합입니다. 즉, 여러 스레드가 동시에 접근해도 연산이 중단되지 않고 완전하게 수행됨을 보장합니다.
*   **동작 원리:** 내부적으로 CAS (Compare-And-Swap) 연산을 사용하여 락 없이(Lock-Free) 동시성을 처리합니다. 현재 값과 예상 값을 비교하여 같으면 새로운 값으로 업데이트하고, 아니면 재시도합니다.
*   **장점:**
    *   락을 사용하지 않으므로 스레드 블로킹이 발생하지 않아 높은 처리량과 낮은 오버헤드를 가집니다.
    *   경쟁이 심한 환경에서 `synchronized`나 `Lock`보다 성능이 우수할 수 있습니다.
    *   단일 변수에 대한 원자적 연산이 필요한 경우 간편하게 사용할 수 있습니다.
*   **단점:**
    *   복잡한 다중 변수 연산이나 코드 블록 전체에 대한 동기화에는 적합하지 않습니다.
    *   CAS 연산의 특성상 반복적인 재시도가 발생할 수 있어, 특정 상황에서는 성능 이점이 크지 않을 수 있습니다.

#### 2.4. `volatile` 키워드

*   **핵심 정의:** 변수의 값을 메인 메모리에서 직접 읽고 쓰도록 강제하여, 스레드 간의 가시성(Visibility)을 보장하는 키워드입니다.
*   **동작 원리:** `volatile` 변수에 대한 읽기/쓰기 작업은 항상 메인 메모리에서 이루어지며, CPU 캐시를 사용하지 않거나 캐시를 즉시 동기화합니다. 이를 통해 한 스레드의 변경 사항이 다른 스레드에게 즉시 보이도록 합니다.
*   **장점:**
    *   `synchronized`에 비해 오버헤드가 매우 적습니다.
    *   단순한 플래그 변수나 상태 변수의 가시성을 보장하는 데 효과적입니다.
*   **단점:**
    *   원자성을 보장하지 않습니다. 즉, `volatile` 변수에 대한 `i++`와 같은 복합 연산은 여전히 경쟁 조건에 취약합니다. (읽기-수정-쓰기 작업에는 부적합)
    *   단순한 가시성 보장 외의 복잡한 동기화 로직에는 사용할 수 없습니다.

#### 2.5. 불변 객체 (Immutable Object)

*   **핵심 정의:** 객체가 생성된 후에는 내부 상태를 변경할 수 없는 객체입니다.
*   **동작 원리:** 객체의 모든 필드를 `final`로 선언하고, 생성자를 통해서만 값을 초기화하며, setter 메서드를 제공하지 않습니다.
*   **장점:**
    *   객체의 상태가 변경되지 않으므로 여러 스레드가 동시에 접근해도 안전합니다. 별도의 동기화 메커니즘이 필요 없습니다.
    *   스레드 안전성을 쉽게 확보할 수 있으며, 코드의 예측 가능성을 높입니다.
    *   캐싱에 유리하고, 해시 기반 컬렉션에서 키로 사용하기 좋습니다.
*   **단점:**
    *   객체의 상태를 변경해야 할 경우, 새로운 객체를 생성해야 하므로 가비지 컬렉션 오버헤드가 발생할 수 있습니다.
    *   모든 객체를 불변으로 만들기는 어려울 수 있습니다.

### 3. 실무에서 주의할 점

*   **최소한의 락 범위:** 락의 범위를 최소화하여 병렬성을 최대한 확보해야 합니다. 불필요하게 넓은 범위에 락을 걸면 성능 저하를 초래합니다.
*   **교착 상태 방지:** 락 순서 규칙을 정하거나, 타임아웃을 사용하는 등 교착 상태를 방지하기 위한 전략을 수립해야 합니다.
*   **성능 고려:** 동기화 기법은 오버헤드를 발생시키므로, 애플리케이션의 특성과 성능 요구사항을 고려하여 적절한 기법을 선택해야 합니다. `Atomic` 클래스는 락 없는 동기화로 성능 이점을 제공할 수 있습니다.
*   **가시성 문제 인지:** `volatile` 키워드가 원자성을 보장하지 않는다는 점을 명확히 인지하고, 복합 연산에는 `synchronized`나 `Atomic` 클래스를 사용해야 합니다.
*   **불변 객체 활용:** 가능한 경우 불변 객체를 적극적으로 활용하여 동시성 문제를 근본적으로 회피하는 것이 가장 좋은 방법입니다.
*   **`java.util.concurrent` 패키지 활용:** 고수준의 동시성 유틸리티(예: `ConcurrentHashMap`, `ExecutorService`)를 사용하여 복잡한 동시성 로직을 직접 구현하는 대신 검증된 솔루션을 활용하는 것이 좋습니다.
